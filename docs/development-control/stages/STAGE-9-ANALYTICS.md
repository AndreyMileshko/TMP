# Stage 9 Manifest — Analytics

## Source

Основные источники требований:

- `docs/TMP/TMP_Initial_Documents/architecture/14-Analytics/Analytics-Specification.md`;
- Warehouse Public API;
- Capability Engine Public API;
- Security Public API;
- Order Management Public API;
- UI/UX Specification;
- Database Specification;
- TMP Constitution;
- принятые ADR.

Analytics использует только публичные read-only контракты других Capability.

---

## Stage objective

Реализовать простой read-only Capability для анализа движения материалов.

Результатом Stage 9 является один пользовательский отчёт:

```text
Аналитика → Движение материалов

```

Отчёт должен позволять:

- просматривать приход материалов;
- просматривать расход материалов;
- просматривать внутренние перемещения;
- просматривать изменения состояния запасов;
- рассчитывать остатки на начало и конец периода;
- фильтровать движения;
- открывать детали движения;
- экспортировать результат в XLSX.

Analytics не изменяет состояние системы и не является источником истины для складских данных.

---

## Stage dependencies

Stage 9 начинается только после завершения:

- Stage 3 — Capability Engine;
- Stage 4 — Security;
- Stage 5 — Order Management;
- Stage 6 — Warehouse;
- Stage 7 — Production.

Stage 8 — Cutting Optimization не является функциональной зависимостью Analytics MVP, но должен быть завершён согласно общей последовательности реализации.

---

## Allowed implementation

Разрешается:

- создать отдельный модуль Analytics;
- подключить модуль к Maven Reactor;
- зарегистрировать Analytics как Capability;
- зарегистрировать экран Analytics;
- зарегистрировать права просмотра и экспорта;
- использовать Warehouse Public API;
- использовать read-only ссылки на Customer Order и Order Item;
- добавить минимальные read-only query-контракты в Warehouse Public API, если их недостаточно для согласованного отчёта;
- реализовать серверную фильтрацию;
- реализовать серверную сортировку;
- реализовать серверную пагинацию;
- реализовать серверную агрегацию;
- реализовать просмотр деталей движения;
- реализовать экспорт в XLSX;
- добавить unit-, integration- и UI-тесты;
- добавить необходимые индексы внутри Warehouse, если их необходимость подтверждена запросами отчёта.

Добавление read-only контрактов в Warehouse не должно изменять складскую бизнес-логику.

---

## Forbidden implementation

Запрещается:

- изменять складские документы из Analytics;
- изменять `Warehouse Movement`;
- изменять `Stock Position`;
- обращаться напрямую к таблицам другого Capability;
- обращаться напрямую к репозиториям другого Capability;
- использовать внутренние классы реализации другого Capability;
- создавать отдельное постоянное аналитическое хранилище;
- создавать копии складских движений;
- публиковать Domain Events Analytics;
- фиксировать открытие отчёта;
- фиксировать изменение фильтров;
- фиксировать экспорт;
- реализовывать финансовую аналитику;
- рассчитывать стоимость или себестоимость материалов;
- реализовывать производственную аналитику;
- реализовывать аналитику раскроя;
- реализовывать прогнозирование;
- реализовывать Dashboard;
- реализовывать диаграммы;
- реализовывать универсальный конструктор отчётов;
- реализовывать сохранённые фильтры;
- реализовывать PDF или CSV;
- расширять Scope Analytics MVP.

## Context restrictions

### Для Domain-задач

Разрешено загружать:

- релевантные разделы Analytics Specification;
- правила классификации движений;
- публичные Warehouse DTO и контракты;
- связанные тесты;
- релевантные принципы Constitution и ADR.

### Для Application-задач

Разрешено загружать:

- конкретный Analytics Use Case;
- Analytics Domain API;
- используемые read-only порты;
- правила транзакций чтения;
- связанные тесты.

### Для Infrastructure-задач

Разрешено загружать:

- конкретный read-only порт;
- публичный Warehouse query-контракт;
- релевантные разделы Database Specification;
- реализацию пагинации, сортировки или агрегации;
- связанные adapter tests.

### Для UI-задач

Разрешено загружать:

- раздел Analytics Specification о пользовательском интерфейсе;
- конкретные Application DTO;
- правила JavaFX и навигации;
- связанные UI-тесты.

### Для Integration-задач

Разрешено загружать:

- публичные контракты Analytics;
- публичные контракты Warehouse;
- публичные контракты Security и Capability Engine;
- acceptance criteria интеграционного сценария;
- инфраструктуру интеграционных тестов.

Загрузка полной реализации Warehouse для обычной задачи Analytics запрещена.

## Required capabilities

Analytics должен зарегистрировать:

```text
analytics.material-movements.view
analytics.material-movements.export

```

Право экспорта не заменяет право просмотра.

Ограничения доступа по отдельным складам в MVP не реализуются.

---

## Required report

Отчёт должен поддерживать фильтры:

- период;
- склад;
- материал;
- категория движения;
- Customer Order;
- Order Item.

По умолчанию используется текущий месяц.

Категории движения:

- `RECEIPT`;
- `CONSUMPTION`;
- `TRANSFER`;
- `STATE_CHANGE`.

Исходный тип Warehouse Operation должен сохраняться и отображаться отдельно от категории Analytics.

---

## Required summary

Сводная часть должна отображать:

- остаток на начало периода;
- приход за период;
- расход за период;
- внутреннее перемещение;
- остаток на конец периода;
- количество операций.

Разные единицы измерения не суммируются.

Количество хранится и рассчитывается через `BigDecimal`.

---

## Required performance behavior

Обязательно:

- серверная фильтрация;
- серверная сортировка;
- серверная пагинация;
- серверная агрегация;
- асинхронная загрузка JavaFX;
- отсутствие загрузки полной выборки для таблицы;
- последовательная запись строк XLSX;
- защита интерфейса от результата устаревшего запроса.

Размер страницы по умолчанию:

```text
100

```

Допустимые размеры:

```text
50
100
200

```

---

## Required verification

Минимальная обязательная проверка Stage 9:

- классификация всех согласованных Warehouse Operation;
- расчёт остатков;
- расчёт прихода и расхода;
- межскладское перемещение;
- резервирование и снятие резерва;
- разные единицы измерения;
- все фильтры;
- сортировка;
- пагинация;
- права просмотра;
- права экспорта;
- экспорт полной выборки в XLSX;
- пустой результат;
- ошибка Warehouse;
- отсутствие изменений складских данных;
- отсутствие прямого доступа к внутренним данным Warehouse;
- успешный `Maven Verify`;
- успешные архитектурные тесты.

---

## Exit criteria

Stage 9 считается завершённым только если:

- Analytics зарегистрирован как Capability;
- экран «Движение материалов» доступен в приложении;
- все согласованные фильтры работают;
- категории движений определяются корректно;
- сводные показатели рассчитываются корректно;
- таблица использует серверную пагинацию;
- детали движения доступны только для чтения;
- XLSX содержит полную отфильтрованную выборку;
- права просмотра и экспорта работают;
- Analytics не изменяет данные других Capability;
- собственные постоянные таблицы Analytics отсутствуют;
- аудит действий Analytics отсутствует;
- запрещённая функциональность MVP не реализована;
- автоматические проверки проходят;
- блокирующие и критические дефекты отсутствуют;
- контрольные документы обновлены.
