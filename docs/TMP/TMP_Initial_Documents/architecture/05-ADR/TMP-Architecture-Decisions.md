# TMP Architecture Decisions (ADR)

**Document ID:** TMP-005  
**Status:** Accepted  
**Version:** 1.3

---

# 1. Назначение

Данный документ содержит ключевые архитектурные решения (Architecture Decision Records, ADR), принятые при проектировании TOP Manufacturing Platform (TMP).

Каждое решение фиксирует не только принятое правило, но и его контекст, причины выбора и последствия для дальнейшего развития платформы.

Документ является единым реестром архитектурных решений проекта и используется:

- архитекторами;
- разработчиками;
- техническими руководителями;
- при проектировании новых Capability;
- при анализе архитектурных изменений.

---

# 2. Назначение ADR

Architecture Decision Record (ADR) — запись о принятом архитектурном решении.

Каждый ADR описывает:

- проблему;
- контекст;
- принятое решение;
- последствия.

После получения статуса **Accepted** решение считается обязательным для всех новых компонентов платформы.

---

# 3. Статусы решений

| Статус | Описание |
|---------|----------|
| Proposed | Решение предложено и находится на обсуждении |
| Accepted | Решение утверждено и является обязательным |
| Superseded | Решение заменено более новым ADR |
| Deprecated | Решение больше не рекомендуется использовать |

---

# 4. Правила оформления ADR

Каждая запись содержит следующие разделы:

- Идентификатор;
- Название;
- Статус;
- Контекст;
- Решение;
- Последствия;
- Связанные документы.

Нумерация ADR является сквозной и никогда не изменяется.

Если решение заменяется новым, старое получает статус **Superseded**, но не удаляется из документа.

# ADR-001

## Название

Platform Core не содержит бизнес-логики.

### Статус

Accepted

### Контекст

TMP проектируется как модульная платформа, состоящая из независимых Capability.

При размещении бизнес-логики в Platform Core возникает сильная связанность компонентов и нарушение границ ответственности.

### Решение

Platform Core отвечает исключительно за инфраструктуру платформы.

Platform Core предоставляет:

- регистрацию Capability;
- жизненный цикл компонентов;
- маршрутизацию запросов;
- инфраструктурные сервисы;
- доступ к конфигурации;
- управление событиями.

Platform Core не содержит бизнес-правил.

### Последствия

Все бизнес-правила реализуются только внутри соответствующих Capability.

Добавление новой функциональности не требует изменения Platform Core.

# ADR-002

## Название

Capability является основной функциональной единицей TMP.

### Статус

Accepted

### Контекст

Платформа должна позволять независимо развивать различные функциональные области без изменения существующих компонентов.

### Решение

Каждая функциональная область реализуется как отдельная Capability.

Capability:

- имеет собственную область ответственности;
- владеет своими данными;
- публикует собственный API;
- регистрируется через Capability Engine;
- самостоятельно объявляет права и события.

### Последствия

Функциональные области становятся независимыми.

Добавление новой Capability не требует изменения существующих компонентов.

# ADR-003

## Название

Capability взаимодействуют только через публичные API.

### Статус

Accepted

### Контекст

Прямой доступ к внутренним данным другой Capability приводит к высокой связанности и невозможности безопасного развития системы.

### Решение

Ни одна Capability не имеет права изменять внутреннее состояние другой Capability напрямую.

Взаимодействие между Capability выполняется исключительно через публичные контракты владельца данных, разделённые по назначению (согласовано с Constitution принцип 28 и ADR-004):

- изменяющие межмодульные бизнес-операции — только через бизнес-документы Document Engine;
- синхронное чтение чужих данных — только через Public Query API владельца;
- асинхронное информирование о завершённых изменениях — только через Domain Events владельца.

Public Query API и Domain Events не могут использоваться для обхода document-driven изменения состояния. Изменяющие application-операции владельца не являются внешним Public API.

### Последствия

Каждая Capability полностью контролирует собственное состояние.

Изменение внутренней реализации не влияет на остальные части платформы.

Границы «чтение / изменение / уведомление» единообразны во всех спецификациях Capability.

### Связанные документы

- Platform-Core-Specification.md
- Capability-Engine-Specification.md
- TMP-Constitution.md

# ADR-004

## Название

Бизнес-документ является точкой входа бизнес-процесса.

### Статус

Accepted

### Контекст

Большинство операций платформы инициируется действиями пользователя и должно иметь полную историю изменений.

### Решение

Любой бизнес-процесс начинается с создания бизнес-документа.

Документ:

- содержит бизнес-смысл операции;
- проходит жизненный цикл;
- инициирует выполнение бизнес-логики;
- является источником пользовательской истории.

Изменяющие бизнес-операции выполняются только через проведение бизнес-документа. Public Query API (синхронное чтение) и Domain Events (асинхронное уведомление о завершённых изменениях) дополняют документную модель, но не могут использоваться для обхода document-driven изменения состояния (согласовано с Constitution принцип 28 и ADR-003).

### Последствия

История действий пользователя полностью сохраняется.

Выполнение процессов становится воспроизводимым и прослеживаемым.

### Связанные документы

- Document-Engine-Specification.md
- TMP-Constitution.md

# ADR-005

## Название

Warehouse является единственным владельцем складских остатков.

### Статус

Accepted

### Контекст

Складские остатки используются множеством функциональных областей платформы:

- Production;
- Procurement;
- Order Management;
- Inventory;
- Logistics.

Если несколько Capability смогут изменять остатки напрямую, неизбежно возникнут рассинхронизация данных, нарушение бизнес-правил и невозможность гарантировать корректность учета.

### Решение

Warehouse является единственным владельцем складских остатков.

Любое изменение запасов выполняется только через Warehouse.

Ни одна Capability не имеет права напрямую изменять:

- Stock Position;
- Warehouse Movement;
- Storage Cell;
- Stock State;
- Batch.

### Последствия

- централизованный контроль складского учета;
- единая история изменений;
- отсутствие конкурирующих механизмов изменения остатков.

### Связанные документы

- Warehouse-Specification.md

# ADR-006

## Название

Stock Position является текущим состоянием склада.

### Статус

Accepted

### Контекст

Необходимо разделить историю операций и текущее состояние склада.

### Решение

Stock Position хранит только текущее состояние запаса.

Stock Position определяется сочетанием:

- Material;
- Warehouse;
- Storage Cell;
- Stock State;
- Batch (при необходимости).

История изменений в Stock Position не хранится.

### Последствия

История и текущее состояние полностью разделены.

Обновление текущего остатка не требует анализа всей истории движений.

### Связанные документы

- Warehouse-Specification.md

# ADR-007

## Название

Warehouse Movement является неизменяемой записью истории.

### Статус

Accepted

### Контекст

История складских операций должна быть полностью прослеживаемой.

Изменение исторических записей делает невозможным проведение аудита и расследование ошибок.

### Решение

Warehouse Movement:

- создается только Warehouse;
- никогда не изменяется;
- никогда не удаляется.

Исправление ошибок выполняется только созданием новых Warehouse Movement.

### Последствия

История склада становится полностью воспроизводимой.

### Связанные документы

- Warehouse-Specification.md

# ADR-008

## Название

Любое изменение запасов выполняется через Warehouse Operation.

### Статус

Accepted

### Контекст

Разные бизнес-документы выполняют различные операции, но все они изменяют складские остатки.

### Решение

Все складские документы преобразуются в универсальный объект Warehouse Operation.

Warehouse выполняет только Warehouse Operation.

Тип документа значения не имеет.

### Последствия

Единый механизм выполнения всех складских операций.

Минимизация дублирования логики.

### Связанные документы

- Warehouse-Specification.md

# ADR-009

## Название

Отрицательные остатки запрещены.

### Статус

Accepted

### Контекст

Отрицательные остатки приводят к недостоверному складскому учету.

### Решение

Перед выполнением Warehouse Operation Warehouse проверяет доступное количество.

При недостатке материалов операция отклоняется.

Warehouse Movement не создаются.

### Последствия

Складской учет всегда остается корректным.

# ADR-010

## Название

Передача материалов на склад производства выполняется в два этапа.

### Статус

Accepted

### Контекст

Передача материалов между складами не гарантирует моментального получения.

Необходимо учитывать материалы, находящиеся в пути.

### Решение

Передача на склад производства состоит из двух этапов:

1. Отгрузка.
2. Приемка.

После отгрузки материалы получают состояние `IN_TRANSIT`.

После подтверждения приемки материалы становятся доступны на складе производства.

### Последствия

Появляется возможность контролировать материалы, находящиеся в пути.

### Связанные документы

- Warehouse-Specification.md

# ADR-011

## Название

Резервирование является изменением состояния Stock Position.

### Статус

Accepted

### Контекст

Резервирование не должно создавать отдельную сущность хранения.

### Решение

Резервирование выполняется переводом количества:

```text

        │
        ▼
RESERVED
```

Отдельная таблица резервов не используется.

### Последствия

Резервирование становится обычной складской операцией.

# ADR-012

## Название

Batch входит в идентификацию Stock Position.

### Статус

Accepted

### Контекст

Для части материалов требуется обязательный партийный учет.

### Решение

# ADR-014

## Название

Warehouse может разделять документ перемещения.

### Статус

Accepted

### Контекст

Иногда только часть строк документа может быть выполнена.

### Решение

Если документ невозможно выполнить полностью, Warehouse может предложить разделение.

Создаются:

- документ с исполнимыми строками;
- документ с проблемными строками.

Исходный документ получает статус `SPLIT`.

### Последствия

Частичное проведение не требуется.

Пользователь получает возможность продолжить работу без изменения исходного документа.

# ADR-016

## Название

В версии 1.0 используется пессимистическая блокировка.

### Статус

Accepted

### Контекст

На начальном этапе эксплуатации важнее простота и предсказуемость работы склада, чем максимальная масштабируемость.

### Решение

Во время выполнения Warehouse Operation необходимые Stock Position блокируются.

После завершения операции блокировки снимаются.

Архитектура должна позволять заменить механизм конкурентного доступа в будущих версиях без изменения публичного API.

### Последствия

Упрощается реализация и исключаются конфликты изменения остатков.

В дальнейшем возможен переход на оптимистическую блокировку без изменения внешнего поведения Warehouse.

# ADR-017

## Название

Позиция заказа является главным производственным объектом платформы.

### Статус

Accepted

### Контекст

Большинство ERP-систем используют заказ как основной объект производственного управления.

Однако в TMP различные изделия одного заказа могут:

- иметь разные спецификации;
- запускаться в производство независимо;
- выпускаться независимо;
- иметь собственные карты раскроя;
- иметь различные производственные статусы.

Использование заказа как основной единицы приводит к усложнению архитектуры и невозможности независимого управления изделиями.

### Решение

Главным производственным объектом TMP является **позиция заказа (Order Item)**.

Все производственные Capability работают исключительно с позициями заказа.

Заказ клиента используется только как контейнер коммерческой информации.

### Последствия

- упрощается архитектура Capability;
- исчезает необходимость хранить производственные данные заказа;
- появляется возможность независимого управления изделиями;
- значительно упрощается Production;
- значительно упрощается Warehouse;
- появляется возможность независимого планирования изделий.

### Связанные документы

- Order-Management-Specification.md
- Production-Specification.md
- Warehouse-Specification.md

# ADR-018

## Название

Спецификация позиции заказа является неизменяемой.

### Статус

Accepted

### Контекст

Изменение спецификации после начала работы других Capability приводит к рассинхронизации данных.

Production, Warehouse, Cutting Optimization и будущий Procurement должны использовать единый источник данных.

### Решение

После утверждения Revision позиции заказа её Specification становится **Immutable**.

Изменение существующей Specification запрещено.

Любое изменение изделия выполняется созданием новой Revision позиции заказа либо новой позиции заказа.

### Последствия

- единый источник истины;
- отсутствует синхронизация между Capability;
- отсутствует необходимость создавать копии Specification;
- упрощается аудит;
- упрощается взаимодействие Capability;
- становится возможным безопасное версионирование изделий.

### Связанные документы

- Order-Management-Specification.md
- Production-Specification.md
- Warehouse-Specification.md
- Cutting-Optimization-Specification.md

# ADR-019

## Название

Каждая Capability владеет только собственными бизнес-данными.

### Статус

Accepted

### Контекст

По мере роста платформы возникает риск дублирования информации между Capability.

Например:

- Production может начать хранить складские остатки;
- Warehouse — производственные статусы;
- Order Management — производственные документы;
- Cutting Optimization — складские данные.

Это приводит к сильной связанности платформы.

### Решение

Каждая Capability является единственным владельцем собственных бизнес-данных.

Другие Capability используют их исключительно через Public API или Domain Events.

Capability не должна хранить копии чужих данных без отдельного архитектурного решения.

### Последствия

- отсутствует дублирование данных;
- снижается связанность;
- упрощается сопровождение;
- становится возможным независимое развитие Capability.

### Связанные документы

- TMP-Constitution.md
- Platform-Core-Specification.md

# ADR-020

## Название

Вычисляемые состояния не должны храниться.

### Статус

Accepted

### Контекст

Во многих информационных системах вычисляемые состояния сохраняются в базе данных.

Это приводит к рассинхронизации и необходимости постоянной синхронизации.

### Решение

Если состояние может быть вычислено из первичных данных, оно не хранится.

Например:

- завершённость производства заказа;
- готовность заказа;
- агрегированные статусы;
- аналитические показатели.

Исключения допускаются только отдельными архитектурными решениями.

### Последствия

- уменьшается объём хранимых данных;
- исключается рассинхронизация;
- упрощается архитектура Capability.

### Связанные документы

- Order-Management-Specification.md
- Production-Specification.md

# ADR-021

## Название

История бизнес-событий является неизменяемой.

### Статус

Accepted

### Контекст

Изменение уже проведённых документов приводит к невозможности восстановления фактической истории работы системы.

### Решение

Проведённые документы не изменяются.

# ADR-022

## Название

Capability предоставляет только собственные бизнес-операции.

### Статус

Accepted

### Контекст

При развитии ERP возникает желание использовать Capability как универсальный источник аналитики.

Это приводит к перегрузке публичных API и нарушению границ ответственности.

### Решение

Public API Capability предоставляет только:

- собственные бизнес-операции;
- собственные бизнес-данные.

Capability не обязана предоставлять аналитические функции.

Аналитика реализуется отдельной Capability.

### Последствия

- компактные Public API;
- слабая связанность;
- независимое развитие аналитической подсистемы.

# ADR-023

## Название

Cutting Plan является независимым технологическим документом.

### Статус

Accepted

### Контекст

Производству требуется использовать результаты оптимизации раскроя без изменения спецификации изделия.

Попытка сохранять результаты раскроя внутри Specification или Production приводит к смешению ответственности различных Capability.

### Решение

Cutting Plan является самостоятельным бизнес-документом Capability Cutting Optimization.

Cutting Plan:

- создаётся независимо от Production;
- не изменяет Specification;
- содержит только технологические данные раскроя;
- может иметь несколько Revision;
- используется Production только после утверждения.

### Последствия

- разделяются ответственность Production и Cutting Optimization;
- спецификация изделия остаётся неизменяемой;
- появляется возможность повторного расчёта без изменения изделия.

### Связанные документы

- Cutting-Optimization-Specification.md
- Production-Specification.md

# ADR-024

## Название

Cutting Plan является необязательным элементом производственного процесса.

### Статус

Accepted

### Контекст

Не все изделия требуют оптимизации раскроя.

Платформа должна поддерживать работу производства независимо от использования Cutting Optimization.

### Решение

Использование Cutting Plan не является обязательным.

Если утверждённая карта раскроя отсутствует, Production использует Specification как источник технологических данных.

### Последствия

- Cutting Optimization становится независимым Capability;
- Production сохраняет работоспособность без карт раскроя;
- возможно поэтапное внедрение функциональности раскроя.

### Связанные документы

- Cutting-Optimization-Specification.md
- Production-Specification.md

# ADR-025

## Название

Для длинномерных материалов приоритет имеет утверждённый Cutting Plan.

### Статус

Accepted

### Контекст

После утверждения карты раскроя именно она определяет фактическую потребность в исходном длинномерном материале.

Использование одновременно Specification и Cutting Plan приводит к двойному учёту потребности.

### Решение

Для длинномерных материалов источником технологической потребности является утверждённый Cutting Plan.

Для остальных материалов используется Specification.

Если карта раскроя отсутствует, потребность полностью определяется Specification.

### Последствия

- отсутствует двойной расчёт материалов;
- Warehouse получает единственный источник потребности;
- сохраняется неизменяемость Specification.

### Связанные документы

- Cutting-Optimization-Specification.md
- Production-Specification.md
- Warehouse-Specification.md

# ADR-026

## Название

Cutting Plan является Immutable-документом.

### Статус

Accepted

### Контекст

Изменение утверждённой карты раскроя нарушает воспроизводимость производства и затрудняет аудит.

### Решение

После утверждения Cutting Plan становится неизменяемым.

Любые изменения выполняются только созданием новой Revision.

Предыдущие Revision сохраняются и доступны для просмотра.

### Последствия

- обеспечивается полная прослеживаемость изменений;
- Production всегда работает с конкретной Revision;
- упрощается аудит производственного процесса.

### Связанные документы

- Cutting-Optimization-Specification.md

# ADR-027

## Название

Production использует только утверждённые Revision Cutting Plan.

### Статус

Accepted

### Контекст

Использование черновых карт раскроя может привести к выпуску продукции по неутверждённой технологии.

### Решение

Production использует исключительно Revision Cutting Plan со статусом `APPROVED`.

Переход карты в состояния `IN_USE` и `COMPLETED` осуществляется автоматически на основании Domain Events Production.

### Последствия

- исключается использование неутверждённых карт;
- обеспечивается согласованный жизненный цикл Production и Cutting Optimization.

### Связанные документы

- Cutting-Optimization-Specification.md

# ADR-028

## Название

Capability-owned Business Document Payload.

### Статус

Accepted

### Контекст

Document Engine владеет платформенным жизненным циклом документа и его metadata (`DocumentId`, тип, заголовок, автор, время создания, статус, операции `post`/`unpost`/`close`/`delete`, вызов зарегистрированного `DocumentProcessor`). Однако бизнес-содержимое документа конкретной Capability (например, состав заказа, спецификация позиции) является предметными данными и не должно попадать в Platform Core как произвольный domain JSON, иначе нарушаются границы владения (ADR-019) и типобезопасность.

Проверка фактического контракта Document Engine: `DocumentOperationContext.document()` предоставляет `DocumentMetadata` с `DocumentId` (`id()`); `DefaultDocumentEngine` аннотирован `@Transactional`, вызывает `DocumentProcessor.onPost(...)` внутри той же транзакции, что и изменение статуса документа и запись lifecycle-журнала, а Domain Events публикуются только после commit (`TransactionAfterCommitEventPublisher`). Атомарность processor-изменения и платформенного изменения документа гарантирована этой транзакционной границей.

### Решение

Document Engine владеет lifecycle и metadata документа. Capability владеет строго типизированным business payload своих документов.

1. Business payload не хранится в `DocumentMetadata` и не хранится в Platform Core как произвольный domain JSON.
2. Payload хранится Capability отдельно (собственный persistence port и adapter) и связывается с платформенным документом через `DocumentId`.
3. `DocumentProcessor` получает `DocumentId` из `DocumentOperationContext` и загружает типизированный payload из репозитория своей Capability.
4. Payload имеет версионируемую схему (`PayloadSchemaVersion`) и optimistic locking черновика (`PayloadRevision`).
5. Payload редактируется только внутренними application/UI use cases Capability, пока платформенный документ находится в редактируемом Draft-состоянии; после успешного проведения payload становится immutable.
6. Payload одной Capability нельзя читать или изменять напрямую из другой Capability; доступ к результатам — только через Public Query API и Domain Events владельца.
7. Обработка проведения должна быть идемпотентной (processing record с уникальностью `DocumentId + Operation`).
8. Атомарность изменения агрегата, записи processing record и изменения результата проведения обеспечивается транзакционной границей Document Engine (processor вызывается внутри транзакции проведения; события — после commit). Если для какой-либо Capability этот контракт окажется недостаточным, требуется отдельный prerequisite-ADR/задача до реализации её Document Processors.

### Последствия

- сохраняется типобезопасность и владение предметными данными (ADR-019);
- Platform Core остаётся свободным от предметного JSON;
- обеспечивается воспроизводимость, идемпотентность и неизменяемость проведённого содержимого;
- межмодульное взаимодействие остаётся document-driven (ADR-003, ADR-004, Constitution принцип 28).

### Связанные документы

# 6. Правила внесения изменений

При изменении архитектуры TMP необходимо соблюдать следующие правила.

1. Если изменяется существующее архитектурное решение, соответствующий ADR получает статус **Superseded**.

2. Новое решение оформляется отдельным ADR с новым номером.

3. Номер ADR никогда не изменяется и не используется повторно.

4. Удаление ADR запрещено.

5. Все новые Capability должны быть проверены на соответствие существующим ADR.

6. Если новое архитектурное решение противоречит существующему ADR, сначала должен быть создан новый ADR, заменяющий прежний.

> **Architecture Rule**  
> Архитектурные решения являются частью контракта платформы. Нарушение принятого ADR допускается только после его официального пересмотра.

# 7. Принципы использования ADR

Архитектурные решения не должны содержать:

- описание пользовательских интерфейсов;
- описание бизнес-процессов;
- описание структуры базы данных;
- описание конкретной реализации алгоритмов.

Для этих целей используются специализированные спецификации Capability.

Каждый ADR отвечает только на четыре вопроса:

- какую проблему необходимо решить;
- какое решение принято;
- почему принято именно это решение;
- какие последствия оно имеет.
