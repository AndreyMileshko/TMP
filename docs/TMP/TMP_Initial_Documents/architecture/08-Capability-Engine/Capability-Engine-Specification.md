# Capability Engine Specification

**Document ID:** TMP-SPEC-008  
**Status:** Accepted  
**Version:** 1.1

---

# Назначение

Capability Engine является инфраструктурным компонентом TOP Manufacturing Platform (TMP), обеспечивающим регистрацию, подключение и управление функциональными возможностями платформы.

Capability Engine позволяет расширять TMP без изменения Platform Core и без жесткой привязки пользовательских рабочих мест к внутренней структуре функциональных областей.

---

# Определение Capability

Capability — законченная функциональная возможность платформы, объединяющая все необходимые элементы для реализации конкретного пользовательского или системного сценария.

Capability может включать:

- команды;
- пользовательские представления;
- пункты навигации;
- права доступа;
- документы;
- Document Processor;
- настройки;
- обработчики событий;
- отчеты;
- элементы рабочего места;
- публичные сервисы (Public API).

Capability не является отдельным приложением и не обязана соответствовать одной функциональной области.

Каждая Capability является владельцем собственной предметной области и самостоятельно управляет своими бизнес-данными, бизнес-логикой и внутренним состоянием.

---

# Примеры Capability

- просмотр складских остатков;
- создание перемещения материалов;
- формирование карты раскроя;
- выпуск изделий;
- просмотр производственных заказов;
- управление пользователями;
- просмотр аналитического отчета.

---

# Ответственность Capability Engine

Capability Engine отвечает за:

- обнаружение доступных Capability;
- регистрацию Capability;
- проверку уникальности идентификаторов;
- активацию и деактивацию Capability;
- регистрацию предоставляемых командами;
- регистрацию прав;
- регистрацию пользовательских представлений;
- регистрацию документов;
- регистрацию Document Processor;
- регистрацию публичных сервисов Capability;
- регистрацию публикуемых событий;
- передачу данных о доступных Capability подсистеме формирования рабочих мест;
- контроль зависимостей между Capability;
- предоставление каталога зарегистрированных Capability;
- завершение работы Capability при остановке платформы.

---

# Что не входит в Capability Engine

Capability Engine не содержит:

- предметную бизнес-логику;
- логику документов;
- складские правила;
- производственные правила;
- пользовательские права конкретных сотрудников;
- реализацию пользовательских экранов;
- внутреннюю логику функциональных областей;
- данные Capability;
- управление бизнес-состоянием Capability.

# Контракт Capability

Каждая Capability должна предоставлять платформе как минимум:

- уникальный идентификатор;
- наименование;
- версию;
- описание;
- перечень зависимостей;
- перечень предоставляемых прав;
- перечень команд;
- перечень пользовательских представлений;
- сведения о регистрируемых документах и обработчиках;
- сведения о публикуемых событиях;
- сведения о публичных сервисах;
- сведения о настройках;
- методы инициализации и завершения работы.

Конкретный программный контракт определяется на этапе технического проектирования и не является частью данной спецификации.

---

# Жизненный цикл Capability

Capability проходит следующие стадии:

1. обнаружена;
2. проверена;
3. зарегистрирована;
4. инициализирована;
5. активна;
6. остановлена;
7. деактивирована.

Ошибка при инициализации одной Capability не должна приводить к повреждению конфигурации других Capability.

---

# Регистрация

При регистрации Capability Engine должен:

1. проверить уникальность идентификатора;
2. проверить совместимость версии;
3. проверить обязательные зависимости;
4. зарегистрировать права;
5. зарегистрировать команды;
6. зарегистрировать представления;
7. зарегистрировать документы и Document Processor;
8. зарегистрировать публичные сервисы Capability;
9. зарегистрировать публикуемые события;
10. зарегистрировать настройки и обработчики событий;
11. сделать Capability доступной для формирования рабочих мест.

Регистрация считается успешной только после выполнения всех обязательных шагов.

---

# Зависимости

Capability может зависеть:

- от Platform Services;
- от контрактов функциональных областей;
- от других Capability.

Циклические зависимости между Capability запрещены.

Зависимости должны быть объявлены явно.

Capability не должна зависеть от внутренней реализации другой Capability.

Допускается зависимость только от публичного контракта.

---

# Права доступа

Capability определяет перечень прав, необходимых для использования ее функций.

Назначение прав конкретным пользователям выполняется подсистемой Security.

Capability Engine не принимает решения о доступе пользователя, а только предоставляет описание доступных прав и функций.

---

# Рабочие места

Рабочее место пользователя формируется из активных Capability, доступных ему по назначенным правам.

Пользователь не обязан видеть внутренние функциональные области платформы.

Отключение Capability должно удалять ее команды, представления и пункты навигации из рабочих мест без изменения Platform Core.

---

# Документы и Document Processor

Capability может регистрировать тип документа и соответствующий Document Processor.

Для одного типа документа допускается только один активный Document Processor.

Capability Engine передает регистрацию в Document Engine и не выполняет бизнес-логику документа самостоятельно.

---

# Публичный API

Capability может публиковать один или несколько публичных сервисов.

Другие Capability взаимодействуют только через зарегистрированные публичные сервисы.

Прямой доступ к внутренним данным Capability запрещен.

Capability Engine регистрирует только описание публичного API и не управляет его реализацией.

---

# События

Capability может публиковать собственные события.

Capability Engine регистрирует описание публикуемых событий.

Публикация и обработка событий выполняется инфраструктурой Event Engine.

Capability Engine не интерпретирует содержимое событий.

---

# Активация и деактивация

Деактивация Capability:

- запрещает создание новых операций Capability;
- удаляет ее функции из рабочих мест;
- не удаляет ранее созданные бизнес-данные и документы;
- не должна нарушать целостность данных;
- допускается только после проверки зависимостей.

Физическое удаление данных Capability не входит в стандартный процесс деактивации.

---

# Ограничения

- Capability не может изменять Platform Core.
- Capability не может напрямую изменять данные другой Capability или функциональной области.
- Capability не может обходить публичный API другой Capability.
- Capability не может обходить Security.
- Capability не может проводить документы без Document Engine.
- Capability не может скрывать свои зависимости.
- Capability не может регистрировать идентификаторы, уже используемые другой Capability.
- Capability не может публиковать незарегистрированные права, документы или события.
