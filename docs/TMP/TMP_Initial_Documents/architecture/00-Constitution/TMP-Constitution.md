# TMP Constitution

**Status:** Accepted  
**Version:** 1.2

## Принципы

1. Platform Core не содержит предметной бизнес-логики.
2. Любое изменение состояния предприятия выполняется только через проведение документа.
3. Все взаимодействие между бизнес-модулями осуществляется только через документы.
4. Документ является объектом платформы, а не отдельного модуля.
5. Заказ клиента является контейнером.
6. Позиция заказа является главным производственным объектом.
7. Каждый модуль отвечает только за собственное состояние позиции заказа.
8. Модули не изменяют данные друг друга напрямую.
9. Бизнес-логика реализуется только на Java.
10. Рабочие места формируются динамически на основании возможностей и прав.
11. Списание материалов выполняется только со склада производства.
12. Все изменения должны быть аудируемыми.
13. Архитектура развивается без изменения Platform Core.
14. Любое решение проверяется на соответствие Конституции.
15. Каждая Capability является единственным владельцем своих бизнес-данных и бизнес-состояния.
16. Изменение бизнес-состояния Capability выполняется только через проведение бизнес-документа (см. принцип 28). Публичный API Capability для других модулей ограничен чтением (Public Query API) и подпиской на завершённые изменения (Domain Events) и не является каналом прямого изменения чужого состояния.
17. История бизнес-операций является неизменяемой. Исправление ошибок выполняется созданием новых документов или компенсирующих операций.
18. Каждая бизнес-сущность имеет только одного владельца. Владелец отвечает за хранение данных, изменение состояния, проверки, публикацию событий и обеспечение целостности данных.
19. Любая бизнес-операция выполняется полностью либо не выполняется вовсе, если иное явно не определено спецификацией соответствующей Capability.
20. Каждая Capability самостоятельно реализует внутреннюю бизнес-логику и не допускает обхода своих бизнес-правил другими Capability.
21. Любое изменение данных должно выполняться владельцем этих данных. Прямой доступ к внутренним структурам другой Capability запрещен.
22. Архитектурные решения, влияющие на платформу в целом, должны быть зафиксированы в TMP-Architecture-Decisions (ADR) до начала реализации.
23. Новые Capability должны проектироваться в соответствии с Конституцией TMP. При невозможности соблюдения одного или нескольких принципов сначала должна быть изменена Конституция или принят соответствующий ADR.
24. Каждая Capability должна иметь четко определенную область ответственности. Пересечение ответственности между Capability не допускается.
25. Внутренняя реализация Capability может изменяться без влияния на другие Capability при сохранении публичного контракта.
26. Все долгоживущие бизнес-данные имеют единственный источник истины (Single Source of Truth). Дублирование данных допускается только для целей кэширования, поиска или аналитики и не может использоваться как источник изменения состояния.
27. Архитектура платформы должна обеспечивать возможность независимого развития, тестирования и сопровождения каждой Capability.
28. Единая формулировка межмодульного взаимодействия:
    - Все изменяющие межмодульные бизнес-операции выполняются только через бизнес-документы.
    - Синхронное чтение данных другой Capability выполняется только через её Public Query API.
    - Асинхронное информирование о завершённых изменениях выполняется через Domain Events.
    - Public Query API и Domain Events не могут использоваться для обхода document-driven изменения состояния.
