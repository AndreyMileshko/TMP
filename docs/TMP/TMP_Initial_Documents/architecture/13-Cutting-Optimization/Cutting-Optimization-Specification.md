# Cutting Optimization Specification

**Document ID:** TMP-SPEC-013  
**Status:** Accepted  
**Version:** 1.0

---

# Назначение

Cutting Optimization является платформенным Capability TOP Manufacturing Platform (TMP), предназначенным для создания, хранения, утверждения и сопровождения карт раскроя длинномерных материалов.

Capability не выполняет производственные операции и не управляет оборудованием.

Cutting Optimization отвечает исключительно за определение оптимального способа использования исходного материала.

---

# Ответственность Capability

Cutting Optimization отвечает за:

- создание карт раскроя;
- автоматическую оптимизацию раскроя;
- полностью ручное создание карт;
- ручную корректировку карт;
- хранение карт раскроя;
- версионирование карт;
- утверждение карт;
- экспорт карт;
- печать карт;
- жизненный цикл карт;
- интеграцию с Production и Warehouse посредством утверждённых карт.

---

# Не входит в ответственность

Capability не отвечает за:

- выполнение производственных операций;
- регистрацию фактически выполненных операций;
- управление оборудованием;
- передачу заданий на пилы;
- получение данных от оборудования;
- управление производственными сменами;
- управление производственными заданиями;
- хранение складских остатков;
- резервирование материалов на складах;
- изменение спецификаций изделий;
- управление заказами.

---

# Основные принципы

1. Cutting Plan является самостоятельным бизнес-документом.
2. Cutting Plan не является частью спецификации изделия.
3. Спецификация изделия никогда не изменяется Cutting Optimization.
4. После утверждения карта становится Immutable.
5. Все изменения выполняются только созданием новой Revision.
6. Одна карта строится только для одного материала (артикул + цвет).
7. Карта раскроя является необязательной. Производство должно корректно работать без неё.
8. При отсутствии утверждённой карты Production использует спецификацию изделия.
9. Cutting Optimization не управляет производством и оборудованием.
10. Capability взаимодействует с другими модулями только через Public API и Domain Events.

---

# Бизнес-документ

Главным документом Capability является:

```text
Cutting Plan
```

Каждый Cutting Plan представляет собой карту раскроя одной производственной партии по одному материалу.

---

# Производственная партия

Карта раскроя всегда создаётся под конкретную производственную партию.

Производственная партия определяется пользователем при создании карты и содержит:

- выбранные заказы;
- выбранные позиции заказов;
- количество изделий каждой позиции;
- один материал (артикул + цвет).

Пример:

```text
Заказ 101
Позиция 1
40 изделий

Заказ 104
Позиция 2
35 изделий

Заказ 108
Позиция 5
25 изделий

Материал:
VEKA 103.211 White
```

Именно эта партия используется при построении карты раскроя.

После полного изготовления данной партии карта считается полностью отработанной.

---

# Создание карты раскроя

Пользователь выполняет следующую последовательность действий:

1. Выбирает необходимые заказы.
2. Отмечает необходимые позиции заказов.
3. Указывает количество изделий каждой позиции, включаемое в производственную партию.
4. Выбирает материал (артикул + цвет).

При выборе материала пользователь должен видеть общий список длинномерных материалов выбранных позиций заказа с группировкой по:

- артикулу;
- цвету.

После выбора материала создаётся отдельная карта раскроя.

В одну карту могут входить позиции из различных заказов при условии использования одного материала.

---

# Один материал в одной карте

Каждая карта раскроя может содержать только один материал.

Под материалом понимается уникальная комбинация:

- артикул;
- цвет.

Например:

```text
VEKA 103.211 White
```

или

```text
Alutech ALT W72 Silver
```

Смешивание различных материалов в одной карте запрещается.

Для другого материала создаётся отдельная карта.

---

# Способы создания карты

Capability поддерживает два равноправных способа создания.

## Автоматическое создание

Пользователь:

- выбирает производственную партию;
- выбирает материал;
- задаёт параметры;
- запускает автоматическую оптимизацию.

После выполнения расчёта создаётся карта раскроя.

---

## Полностью ручное создание

Пользователь может создать карту без использования алгоритма оптимизации.

При этом пользователь самостоятельно:

- создаёт исходные хлысты;
- размещает отрезки;
- формирует раскладку;
- сохраняет карту.

Автоматический расчёт при этом не используется.

Оба способа являются полностью поддерживаемыми сценариями Capability.

---

# Ручное редактирование карты

После создания карты пользователь может свободно выполнять ручную корректировку.

Разрешается:

- изменять расположение Cut Piece;
- изменять состав Source Bar;
- изменять последовательность резов;
- добавлять исходные хлысты;
- удалять пустые исходные хлысты;
- выполнять повторную автоматическую оптимизацию;
- изменять параметры расчёта.

Все изменения допускаются только для карты в статусе `DRAFT`.

После утверждения любые изменения запрещены.

---

# Проверка карты

Перед утверждением Cutting Optimization обязан проверить:

- полноту обязательных Cut Piece;
- отсутствие дублирования обязательных Cut Piece;
- математическую корректность карты;
- непревышение длины каждого исходного хлыста;
- корректность ширины реза;
- отсутствие смешивания различных материалов;
- корректность производственной партии;
- отсутствие конфликтов с другими утверждёнными картами.

При обнаружении ошибок утверждение невозможно.

---

# Параметры расчёта

Каждая карта содержит неизменяемый набор параметров расчёта.

Parameters включают:

- стратегию оптимизации;
- ширину реза;
- минимальный технологический остаток;
- допустимые длины исходных хлыстов;
- количество исходных хлыстов каждой длины;
- режим ограничения количества хлыстов;
- версию алгоритма оптимизации.

После утверждения параметры становятся неизменяемыми.

---

# Стратегии оптимизации

Capability поддерживает различные стратегии оптимизации.

В версии 1.0 предусматривается выбор одной стратегии при создании карты.

Например:

- минимальные отходы;
- минимальное количество исходных хлыстов;
- технологический компромисс.

Алгоритм оптимизации должен быть расширяемым.

---

# Использование складских остатков

При создании карты раскроя складские остатки не используются алгоритмом.

Пользователь должен видеть информацию о наличии материалов на складе производства, однако она не влияет на расчёт.

Все параметры формирования карты определяются пользователем независимо от текущих складских остатков.

---

# Исходные хлысты

Source Bar представляет собой один исходный длинномерный материал.

Source Bar содержит:

- длину исходного материала;
- порядковый номер;
- список размещённых Cut Piece;
- остаток;
- коэффициент использования.

Source Bar является структурным элементом Cutting Plan и не является самостоятельным бизнес-объектом платформы.

---

# Cut Piece

Cut Piece является основной вычислительной единицей карты.

Каждый Cut Piece содержит:

- ссылку на Order Item;
- ссылку на Revision позиции заказа;
- ссылку на строку спецификации;
- материал;
- длину;
- порядковый номер внутри Source Bar;
- ссылку на Source Bar;
- технологические параметры (при необходимости).

Именно с объектами Cut Piece работают алгоритмы оптимизации.

Пользователь напрямую с объектами Cut Piece не работает.

---

# Calculation Result

Calculation Result хранит агрегированные результаты построения карты.

Например:

- количество исходных хлыстов;
- суммарную длину исходного материала;
- суммарную длину отходов;
- коэффициент использования материала;
- количество резов;
- другие агрегированные показатели расчёта.

Calculation Result формируется автоматически после выполнения расчёта.

---

# Архитектура модели данных

Модель данных Cutting Optimization строится вокруг следующих объектов:

```text
Cutting Plan
│
├── Parameters
├── Batch Allocation
├── Source Bar
│      │
│      └── Cut Piece
│
└── Calculation Result
```

## Cutting Plan

Главный бизнес-документ Capability.

Хранит:

- материал;
- Revision;
- статус;
- параметры;
- производственную партию;
- результат расчёта;
- историю изменений.

---

## Batch Allocation

Batch Allocation определяет производственную партию карты.

Для каждой позиции заказа хранится:

- Order Item;
- Revision;
- количество изделий;
- связанные строки спецификации;
- связанные Cut Piece.

Batch Allocation определяет состав производственной партии, для которой построена карта раскроя.

---

## Две модели Capability

Внутри Capability используются две независимые модели.

### Вычислительная модель

Используется алгоритмом оптимизации.

Главным объектом является Cut Piece.

### Пользовательская модель

Используется пользовательским интерфейсом.

Главным объектом является Source Bar.

Пользователь всегда работает с исходными хлыстами, а не с внутренними вычислительными структурами.
# Версионирование

Каждый Cutting Plan поддерживает встроенное версионирование.

Карта имеет постоянный логический идентификатор и последовательность редакций.

Пример:

```text
Cutting Plan №125

├── Revision 1
├── Revision 2
├── Revision 3
└── Revision 4
```

---

## Создание новой Revision

Новая Revision создаётся при любом изменении утверждённой карты.

Например:

- изменение состава производственной партии;
- изменение параметров расчёта;
- изменение стратегии оптимизации;
- повторная автоматическая оптимизация;
- ручное изменение раскладки;
- изменение исходных хлыстов.

Предыдущие Revision никогда не изменяются.

Новая Revision всегда создаётся в статусе `DRAFT`.

---

## Актуальная Revision

Для каждого Cutting Plan существует одна актуальная утверждённая Revision.

Именно она:

- используется Production;
- экспортируется;
- отображается пользователю по умолчанию.

Исторические Revision доступны только для просмотра и анализа.

---

# Жизненный цикл

Cutting Plan имеет собственный жизненный цикл.

```text
DRAFT
   │
   ▼
APPROVED
   │
   ▼
IN_USE
   │
   ▼
COMPLETED
   │
   ▼
ARCHIVED
```

Обратные переходы отсутствуют.

---

## DRAFT

Карта создаётся и редактируется.

Разрешается:

- автоматический расчёт;
- ручное создание;
- ручное редактирование;
- изменение параметров;
- изменение производственной партии;
- повторная оптимизация.

Карта не влияет на работу других Capability.

---

## APPROVED

После утверждения карта становится неизменяемой.

Дополнительно:

- резервируется количество изделий производственной партии;
- карта становится доступной Production;
- карта может экспортироваться.

---

## IN_USE

Означает начало использования карты Production.

Статус устанавливается Capability Cutting Optimization после получения события:

```text
ProductionLaunchStarted
```

---

## COMPLETED

Производственная партия полностью изготовлена.

После перехода:

- карта считается полностью отработанной;
- резервирование количества изделий снимается;
- карта больше не может использоваться в новых документах Production.

---

## ARCHIVED

Карта выведена из активного использования.

При этом сохраняются:

- история;
- аудит;
- экспорт;
- просмотр.

---

# Производственные партии

Карта раскроя создаётся для конкретной производственной партии.

Производственная партия определяется пользователем.

Одна позиция заказа может быть разделена между несколькими производственными партиями.

Например:

```text
Order Item

100 изделий

↓

Cutting Plan №1
40 изделий

↓

Cutting Plan №2
30 изделий

↓

Cutting Plan №3
30 изделий
```

Такой подход позволяет выпускать крупные заказы поэтапно.

---

# Контроль количества изделий

При утверждении карты Capability выполняет проверку.

Суммарное количество изделий одной позиции заказа, включённое в карты со статусами:

- APPROVED;
- IN_USE;

не может превышать количество изделий позиции заказа.

Проверка выполняется отдельно для каждой Revision позиции заказа.

---

# Работа черновиков

Карты в статусе `DRAFT`:

- не резервируют количество изделий;
- не блокируют создание других карт;
- не участвуют в проверке конфликтов;
- могут свободно пересекаться между собой.

Проверка конфликтов выполняется исключительно при утверждении карты.

---

# Работа с Revision позиции заказа

Каждая карта создаётся для конкретной Revision позиции заказа.

Batch Allocation хранит ссылки на:

- Order Item;
- Order Item Revision;
- строки спецификации.

Если создаётся новая Revision позиции заказа:

- существующая карта остаётся корректной для предыдущей Revision;
- новая Revision автоматически не включается ни в одну карту;
- при необходимости создаётся новая Revision Cutting Plan.

---

# Интеграция с Production

Production использует только утверждённые карты раскроя.

Production никогда:

- не создаёт карты;
- не изменяет карты;
- не выполняет оптимизацию;
- не изменяет параметры карты.

При отсутствии утверждённой карты Production использует спецификацию изделия.

---

# Использование карты Production

Для длинномерных материалов:

```text
Production
        │
        ▼
Cutting Plan
```

Для остальных материалов:

```text
Production
        │
        ▼
Item Specification
```

Таким образом Production использует комбинированную модель получения производственной потребности.

---

# Интеграция с Warehouse

Warehouse никогда не работает с объектами Cut Piece.

Warehouse получает только итоговую потребность в исходном длинномерном материале.

Например:

```text
2 хлыста
6500 мм
```

Warehouse не знает:

- расположение деталей;
- последовательность резов;
- структуру Source Bar;
- параметры оптимизации.

Ответственность Warehouse ограничивается обеспечением исходного материала.

# Интеграция с Order Management

Cutting Optimization не изменяет данные Order Management.

Capability использует Order Management исключительно как источник данных.

Используются:

- Customer Order;
- Order Item;
- Revision позиции заказа;
- спецификация изделия.

После создания карты данные спецификации не изменяются.

---

# Источник истины

После утверждения карты источники данных распределяются следующим образом.

## Спецификация изделия

Спецификация является источником истины для:

- состава изделия;
- размеров деталей;
- количества деталей;
- структуры изделия.

---

## Cutting Plan

Cutting Plan является источником истины только для определения потребности в исходном длинномерном материале.

Cutting Plan не изменяет изделие.

Он определяет способ использования исходного материала.

---

# Работа без карты раскроя

Cutting Optimization является необязательным Capability.

Если для производственной партии отсутствует утверждённая карта:

- создание Production Document разрешается;
- выпуск продукции выполняется;
- потребность определяется по спецификации изделия.

Это обеспечивает работоспособность платформы без использования Cutting Optimization.

---

# Экспорт

Cutting Optimization предоставляет единый механизм экспорта карт.

Экспорт является ответственностью Capability.

Поддерживается концепция Export Provider.

```text
Cutting Plan
      │
      ▼
Export Engine
      │
 ┌────┼────┬────┬────┐
 ▼    ▼    ▼    ▼
CSV XML JSON TXT
```

В дальнейшем могут быть добавлены специализированные экспортёры для оборудования различных производителей.

---

# Архитектура экспорта

Cutting Plan никогда не зависит от конкретного формата экспорта.

Модель данных Capability не содержит информации о формате оборудования.

Экспорт выполняется отдельным компонентом Export Provider.

Это позволяет расширять перечень поддерживаемых форматов без изменения модели данных.

---

# Public API

Capability предоставляет следующий Public API.

```text
createCuttingPlan(request)
```

Создание новой карты раскроя.

---

```text
calculateCuttingPlan(planRevisionId)
```

Выполнение автоматической оптимизации.

---

```text
validateCuttingPlan(planRevisionId)
```

Проверка корректности карты.

---

```text
approveCuttingPlan(planRevisionId)
```

Утверждение карты.

---

```text
createCuttingPlanRevision(cuttingPlanId)
```

Создание новой Revision.

---

```text
archiveCuttingPlan(cuttingPlanId)
```

Архивирование карты.

---

```text
getCuttingPlan(cuttingPlanId, revision)
```

Получение утверждённых карт для выбранной производственной партии.

```text
saveCuttingPlanDraft(planRevisionId, draftData)
```

# Domain Events

Capability публикует только события, представляющие интерес для других Capability.

---

## CuttingPlanApproved

Карта утверждена.

После публикации события:

- карта становится доступной Production;
- резервируется количество изделий производственной партии.

---

## CuttingPlanInUse

Публикуется после перехода карты в статус `IN_USE`.

---

## CuttingPlanCompleted

Публикуется после полного изготовления производственной партии.

После публикации снимается резервирование количества изделий.

После получения события:

```text
Production
        │
        ▼
ProductionLaunchCompleted
        │
        ▼
Cutting Optimization
        │
        ▼
CuttingPlanCompleted
```

---

## CuttingPlanArchived

Публикуется после архивирования карты.

---

# Внутренние события

Следующие события не публикуются наружу:

- CuttingPlanCreated;
- CuttingPlanRevisionCreated;
- CuttingPlanCalculated;
- CuttingPlanValidationFailed;
- CuttingPlanDraftSaved;
- CuttingPlanExported.

Они являются внутренними событиями Capability.

---

# Получение Domain Events

Capability подписывается на события других Capability.

В версии 1.0 используется событие:

```text
ProductionLaunchStarted
```

После получения данного события Capability самостоятельно:

- переводит карту в статус `IN_USE`;
- публикует событие `CuttingPlanInUse`.

Ни один Capability не изменяет документы Cutting Optimization напрямую.

Аналогичным образом Capability обрабатывает событие ProductionLaunchCompleted, переводит карту в статус COMPLETED и публикует событие CuttingPlanCompleted.

# Security

Capability регистрирует следующие права доступа.

| Capability | Назначение |
|------------|------------|
| `CUTTING_PLAN_VIEW` | Просмотр карт раскроя |
| `CUTTING_PLAN_CREATE` | Создание карт раскроя |
| `CUTTING_PLAN_EDIT_DRAFT` | Редактирование карт в статусе `DRAFT` |
| `CUTTING_PLAN_APPROVE` | Утверждение карт |
| `CUTTING_PLAN_CREATE_REVISION` | Создание новой Revision |
| `CUTTING_PLAN_ARCHIVE` | Архивирование карт |
| `CUTTING_PLAN_EXPORT` | Экспорт карт |
| `CUTTING_PLAN_PRINT` | Печать карт |

---

# Правила регистрации Capability

Security защищает только бизнес-операции.

Capability не регистрирует отдельные права для внутренних действий редактора карты.

Например, не регистрируются отдельные права для:

- перемещения Cut Piece;
- добавления Source Bar;
- удаления Source Bar;
- изменения порядка размещения.

Подобные действия являются частью процесса редактирования карты и относятся к праву `CUTTING_PLAN_EDIT_DRAFT`.

---

# Audit

Capability ведёт аудит только значимых бизнес-событий.

Редактирование карты в статусе `DRAFT` не журналируется.

---

## Аудируемые события

Подлежат регистрации:

- CuttingPlanCreated;
- CuttingPlanRevisionCreated;
- CuttingPlanApproved;
- CuttingPlanInUse;
- CuttingPlanCompleted;
- CuttingPlanArchived;
- CuttingPlanExported.

---

## Не аудируются

Не фиксируются отдельными записями:

- изменение параметров расчёта;
- изменение порядка Cut Piece;
- изменение Source Bar;
- ручная корректировка;
- повторная оптимизация;
- промежуточные сохранения черновика.

История изменений обеспечивается механизмом Revision.

---

# Invariants

Следующие правила являются обязательными для Capability.

---

## Invariant 1

Утверждённая карта раскроя является неизменяемой.

---

## Invariant 2

Каждая карта относится только к одной Revision каждой включённой позиции заказа.

---

## Invariant 3

Одна карта содержит только один материал.

Материал определяется комбинацией:

- артикул;
- цвет.

---

## Invariant 4

Одна карта может содержать позиции из нескольких заказов.

---

## Invariant 5

Все Cut Piece карты относятся только к одному материалу.

---

## Invariant 6

Количество изделий, включённое в карты со статусами:

- APPROVED;
- IN_USE;

не может превышать количество изделий соответствующей Revision позиции заказа.

---

## Invariant 7

Карты в статусе `DRAFT`:

- не резервируют количество изделий;
- не участвуют в проверке конфликтов;
- не блокируют создание других карт.

---

## Invariant 8

Каждый обязательный Cut Piece может входить только в одну активную карту раскроя.

---

## Invariant 9

Production никогда не изменяет содержимое Cutting Plan.

---

## Invariant 10

Cutting Optimization никогда не изменяет спецификацию изделия.

---

# Architecture Rules

## Rule 1

Cutting Optimization является единственным владельцем документа Cutting Plan.

Ни один другой Capability не имеет права изменять данный документ напрямую.

---

## Rule 2

Все изменения утверждённой карты выполняются только созданием новой Revision.

Редактирование утверждённой карты запрещено.

---

## Rule 3

Production использует Cutting Plan исключительно как источник технологических данных.

Production никогда:

- не создаёт карту;
- не изменяет карту;
- не пересчитывает карту;
- не выполняет оптимизацию.

---

## Rule 4

Warehouse никогда не работает с объектами Cut Piece.

Warehouse получает только итоговую потребность в исходном длинномерном материале.

---

## Rule 5

Specification является единственным источником истины о составе изделия.

Cutting Plan определяет исключительно способ использования исходного материала.

---

## Rule 6

Все взаимодействие между Capability выполняется только посредством Public API и Domain Events.

Запрещается:

- прямой доступ к базе данных другого Capability;
- вызов внутренних сервисов другого Capability;
- изменение объектов другого Capability.

---

## Rule 7

Cutting Optimization не содержит логики производства.

Capability отвечает исключительно за:

- расчёт;
- хранение;
- редактирование;
- утверждение;
- экспорт;
- жизненный цикл карт раскроя.

---

## Rule 8

Cutting Engine является расширяемой архитектурной точкой платформы.

В версии 1.0 реализуется только алгоритм линейного раскроя.

Архитектура должна позволять в дальнейшем подключать:

- Sheet Optimizer;
- Glass Optimizer;
- Panel Optimizer;
- Custom Optimizer.

Без изменения модели Cutting Plan.

---

# Version 1.0 Limitations

В версию 1.0 входят:

- линейный раскрой;
- автоматическая оптимизация;
- полностью ручное создание карт;
- ручная корректировка;
- версионирование;
- производственные партии;
- экспорт карт;
- интеграция с Production;
- интеграция с Warehouse;
- интеграция с Security.

---

В версию 1.0 не входят:

- листовой раскрой;
- раскрой стекла;
- раскрой панелей;
- автоматическое формирование производственных партий;
- автоматический запуск расчёта;
- управление оборудованием;
- получение данных от оборудования;
- регистрация фактически выполненных операций;
- MES-функции;
- автоматический выбор оборудования;
- оптимизация одновременно по нескольким материалам;
- совместное редактирование одной карты несколькими пользователями в режиме реального времени.

---

# Итог

Capability Cutting Optimization предоставляет единый механизм:

- формирования производственных партий;
- построения карт раскроя;
- оптимизации использования длинномерных материалов;
- хранения и версионирования карт;
- утверждения карт;
- экспорта карт;
- предоставления технологических данных Production.

Capability является независимым платформенным компонентом и взаимодействует с другими Capability исключительно посредством Public API и Domain Events.
