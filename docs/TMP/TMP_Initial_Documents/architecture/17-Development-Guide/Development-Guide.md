# Development Guide

**Document ID:** TMP-GUIDE-001  
**Status:** Accepted  
**Version:** 1.0

# 1. Назначение

Development Guide определяет обязательный процесс разработки для TOP Manufacturing Platform (TMP).

Документ устанавливает:

- процесс разработки;
- подготовку перед реализацией;
- правила работы Cursor AI;
- правила ограничения контекста;
- процедуру Question Gate;
- Definition of Ready (DoR);
- Definition of Done (DoD);
- требования к документации;
- управление изменениями;
- контроль качества.

Настоящий документ является обязательным для каждого разработчика и каждой сессии Cursor AI.

---

# 2. Область применения

Настоящее руководство распространяется на:

- разработку новой функциональности;
- исправление ошибок;
- рефакторинг;
- изменения архитектуры;
- изменения базы данных;
- изменения пользовательского интерфейса;
- изменения инфраструктуры;
- обновление зависимостей;
- разработку с использованием Cursor AI.

---

# 3. Приоритет документов

При возникновении противоречий между документами используется следующий порядок приоритета:

1. Constitution
2. Architecture Decision Record (ADR)
3. Architecture Specifications
4. Database Specification
5. Development Guide
6. Code Quality Standards
7. Master Implementation Plan
8. Stage Implementation Checklist
9. Task Prompt

Cursor AI никогда не должен самостоятельно разрешать противоречия между документами.

Реализация должна быть остановлена до момента устранения конфликта.

---

# 4. Принципы разработки

## 4.1 Поэтапная разработка

Разработка выполняется исключительно по утверждённым этапам реализации.

Запрещается:

- реализовывать функциональность будущих этапов;
- расширять Scope текущей задачи;
- писать код «на будущее»;
- пропускать этапы реализации.

---

## 4.2 Одна задача — одно изменение

Каждая задача должна:

- иметь чётко определённый Scope;
- решать одну логическую задачу;
- изменять только необходимые файлы;
- иметь измеримые критерии завершения.

Крупные задачи должны быть разделены до начала реализации.

---

## 4.3 Архитектура раньше кода

До начала реализации должны быть определены:

- цель;
- Scope;
- затрагиваемые модули;
- архитектурное решение;
- интерфейсы;
- изменения базы данных;
- критерии завершения.

Архитектура никогда не должна формироваться методом проб и ошибок в процессе написания кода.

---

## 4.4 Принцип минимальных изменений

Разрешены только изменения, необходимые для выполнения утверждённой задачи.

Без отдельного согласования запрещается:

- выполнять массовый рефакторинг;
- переименовывать несвязанные классы;
- изменять соседние модули;
- обновлять зависимости;
- изменять архитектуру;
- исправлять несвязанные ошибки;
- форматировать несвязанные файлы.

---

## 4.5 Работоспособность проекта

После завершения каждой задачи проект должен:

- успешно компилироваться;
- проходить обязательные тесты;
- успешно выполнять Maven Verify;
- оставаться работоспособным, если это применимо;
- не содержать временного кода.

---

## 4.6 Архитектура важнее скорости

Скорость разработки никогда не является основанием для:

- нарушения архитектуры;
- прямой связанности модулей;
- размещения бизнес-логики в UI;
- обхода Application API;
- отказа от тестирования;
- использования временных архитектурных решений.

---

## 4.7 Явная фиксация решений

Каждое значимое решение должно быть задокументировано.

В зависимости от типа решения оно фиксируется в:

- ADR;
- Specification;
- Implementation Plan;
- Checklist;
- Implementation Log.

Решения, не зафиксированные в документации, считаются несуществующими.

---

## 4.8 Отсутствие скрытых предположений

Cursor AI никогда не должен самостоятельно предполагать:

- бизнес-правила;
- права доступа;
- значения по умолчанию;
- форматы хранения данных;
- интеграционные контракты;
- архитектурные зависимости.

Если документация не позволяет однозначно определить решение, обязательно используется Question Gate.

---

# 5. Workflow выполнения задачи

Каждая задача проходит одинаковый жизненный цикл.

## Шаг 1 — Получение задачи

Определяются:

- цель;
- Scope;
- затрагиваемые модули;
- этап реализации.

---

## Шаг 2 — Question Gate

До начала реализации должны быть устранены все блокирующие неоднозначности.

До завершения Question Gate запрещается писать Production code.

---

## Шаг 3 — Подготовка контекста

Разрешается загружать только документы, необходимые для выполнения текущей задачи.

Загружать в Cursor AI весь проект запрещается.

---

## Шаг 4 — Реализация

Разрешается:

- создавать необходимые файлы;
- изменять утверждённые файлы;
- писать тесты;
- обновлять связанную документацию.

Запрещается:

- изменять архитектуру;
- изменять несвязанные модули;
- изменять несвязанные настройки.

---

## Шаг 5 — Самопроверка

Перед завершением задачи Cursor AI обязан проверить:

- успешную компиляцию;
- соответствие архитектуре;
- соблюдение стандартов кодирования;
- корректность импортов;
- форматирование;
- отсутствие временного кода;
- прохождение обязательных тестов.

---

## Шаг 6 — Завершение задачи

Перед закрытием задачи необходимо:

- обновить Checklist;
- обновить Implementation Log;
- обновить документацию (при необходимости);
- проверить выполнение критериев завершения.

Только после выполнения всех перечисленных действий задача считается завершённой.
# 6. Правила работы Cursor AI

Cursor AI обязан соблюдать строгие ограничения контекста.

Правила:

- Один чат = одна задача.
- Один Prompt = одна цель.
- Один этап реализации = один Checklist.
- Один модуль реализуется за один раз.
- Загружается только необходимая документация.
- Запрещается анализировать несвязанные модули.
- Запрещается превышать доступное контекстное окно.

Крупные задачи всегда должны разделяться на независимые подзадачи.

---

# 7. Question Gate

Question Gate является обязательным при любой неоднозначности реализации.

---

## 7.1 Обязательные вопросы

Cursor AI обязан остановить реализацию, если:

- существует несколько вариантов реализации;
- отсутствуют бизнес-правила;
- обнаружены противоречия в документации;
- не определён владелец данных;
- не определены границы транзакции;
- требуется новая зависимость;
- требуется изменение публичного API;
- требуется изменение схемы базы данных;
- требуется изменение архитектуры;
- обнаружено расширение Scope;
- отсутствуют критерии завершения.

---

## 7.2 Вопросы, которые запрещено задавать

Запрещается задавать вопросы, если ответ уже содержится в:

- ADR;
- Specification;
- Code Quality Standards;
- утверждённой документации.

Cursor AI обязан самостоятельно принимать обычные технические решения, если они однозначно определяются документацией.

---

## 7.3 Формат вопросов

Каждый вопрос должен содержать:

1. описание неоднозначности;
2. причину блокировки реализации;
3. возможные варианты решения;
4. последствия каждого варианта;
5. рекомендуемый вариант;
6. перечень затрагиваемой документации.

---

## 7.4 Правила остановки

До завершения Question Gate Cursor AI запрещается:

- писать Production code;
- создавать миграции;
- изменять контракты;
- устанавливать зависимости;
- расширять Scope.

Разрешается только анализ и уточнение требований.

---

## 7.5 Результат

Question Gate считается завершённым только после того, как определены:

- утверждённый Scope;
- утверждённое архитектурное решение;
- затрагиваемые модули;
- необходимая документация;
- критерии завершения.

После этого задача получает статус:

**Ready for Implementation**

---

# 8. Definition of Ready (DoR)

Реализация может быть начата только при выполнении следующих условий:

- определена цель;
- определён Scope;
- известны затрагиваемые модули;
- известны файлы, разрешённые к изменению;
- завершён Question Gate;
- документация не содержит противоречий;
- определены критерии завершения;
- определены необходимые тесты;
- определены обязательные проверки;
- утверждён этап реализации.

При невыполнении хотя бы одного условия реализация запрещается.

---

# 9. Definition of Done (DoD)

Задача считается завершённой только при выполнении всех требований.

## Код

- реализация полностью завершена;
- отсутствуют TODO;
- отсутствуют FIXME;
- отсутствуют HACK;
- отсутствует закомментированный код;
- отсутствует неиспользуемый код.

---

## Архитектура

- соблюдены границы модулей;
- соблюдены правила зависимостей;
- отсутствуют циклические зависимости;
- соблюдены требования Architecture Specifications.

---

## Качество

- соблюдены Code Quality Standards;
- выполнено форматирование;
- успешно пройден Static Analysis.

---

## Тестирование

- реализованы необходимые тесты;
- все тесты успешно проходят;
- Maven Verify выполняется без ошибок;
- проект успешно компилируется.

---

## Документация

При необходимости обновлены:

- Specifications;
- ADR;
- Implementation Log;
- Checklists.

---

## Финальная проверка

Cursor AI обязан подтвердить:

- Checklist полностью выполнен;
- отсутствуют известные дефекты;
- Scope реализован полностью;
- код готов к Code Review.

Только после этого задача получает статус:

**Done**

---

# 10. Правила ведения документации

Документация является частью результата разработки.

После завершения каждой задачи Cursor AI обязан определить необходимость её обновления.

При необходимости обновляются:

- Specifications;
- ADR;
- Implementation Plan;
- Master Checklist;
- Stage Checklist;
- Implementation Log.

Код и документация всегда должны находиться в синхронизированном состоянии.
# 11. Управление изменениями

Каждое изменение должно:

- быть отслеживаемым;
- относиться только к одной задаче;
- оставаться в пределах утверждённого Scope.

Дополнительно:

- изменения архитектуры требуют обновления ADR;
- изменения базы данных требуют обновления Database Specification;
- изменения API требуют обновления соответствующих Specifications;
- изменения UI требуют обновления UI/UX Specification.

---

# 12. Правила работы с Git

Каждый Commit должен:

- содержать одно логическое изменение;
- соответствовать одной задаче;
- быть понятным без дополнительного контекста.

Запрещается:

- объединять несвязанные изменения;
- выполнять Commit временного кода;
- выполнять Commit некомпилируемого проекта;
- выполнять Commit неиспользуемого кода.

Перед каждым Commit необходимо убедиться, что:

- проект успешно компилируется;
- все тесты успешно проходят;
- документация обновлена;
- Checklist обновлён;
- выполнен Code Review.

---

# 13. Code Review

Каждая завершённая задача должна проходить Code Review.

Code Review включает проверку:

- архитектуры;
- границ модулей;
- наименований;
- читаемости кода;
- обработки ошибок;
- дублирования логики;
- покрытия тестами;
- отсутствия временного кода;
- соответствия стандартам кодирования.

Даже при индивидуальной разработке Cursor AI обязан выполнить данную проверку перед завершением задачи.

---

# 14. Запрещённые практики

Запрещается:

- выполнять реализацию без утверждённого Scope;
- обходить Question Gate;
- нарушать архитектуру;
- создавать прямую связанность модулей;
- дублировать бизнес-логику;
- использовать Magic Numbers;
- создавать скрытые зависимости;
- использовать глобальное изменяемое состояние без согласования;
- создавать циклические зависимости;
- использовать временные архитектурные решения без документации;
- оставлять TODO/FIXME/HACK в Production code;
- оставлять закомментированный код;
- хранить неиспользуемые классы;
- хранить неиспользуемые зависимости;
- изменять файлы вне текущего Scope.

Любое нарушение настоящих правил считается дефектом процесса разработки.
