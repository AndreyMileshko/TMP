# Code Quality Standards

**Document ID:** TMP-STD-001
**Status:** Accepted
**Version:** 1.0

---

# 1. Назначение

Code Quality Standards определяет обязательные стандарты написания кода для TOP Manufacturing Platform (TMP).

Документ устанавливает единые требования к:

- структуре проекта;
- архитектурным слоям;
- стилю программирования;
- именованию;
- тестированию;
- организации зависимостей;
- правилам работы с Git;
- критериям качества реализуемого кода.

Требования настоящего документа обязательны для:

- разработчиков;
- Cursor AI;
- будущих AI-агентов.

При возникновении противоречий между личными предпочтениями разработчика и настоящим документом приоритет имеет настоящий документ.

---

# 2. Общие правила кодирования

Обязательные требования:

- Один класс — одна ответственность.
- Один метод — одна задача.
- Запрещены скрытые побочные эффекты.
- Запрещено дублирование логики.
- Запрещены Magic Numbers.
- Запрещено использование глобального изменяемого состояния.
- Все зависимости должны быть явными.
- Код должен быть детерминированным.
- Код должен быть читаемым без комментариев.
- Комментарии допускаются только при невозможности выразить смысл кодом.
- Все публичные API должны быть стабильными.
- Любые временные решения запрещены.
- TODO, FIXME и HACK в Production code запрещены.
- Закомментированный код запрещён.
- Неиспользуемый код запрещён.
- Неиспользуемые зависимости запрещены.
- Любое предупреждение компилятора должно быть устранено либо обосновано.

# 3. Naming Convention

## Общие правила

- Все идентификаторы именуются только на английском языке.
- Используются полные слова без необщепринятых сокращений.
- Название должно отражать назначение объекта.
- Запрещены неоднозначные имена.
- Запрещены сокращения, специфичные для проекта.
- Запрещено использовать цифры в именах, если они не являются частью общепринятого стандарта.

---

## Пакеты

- Только lowercase.
- Разделение через точку.
- Имя пакета должно отражать его назначение.

---

## Классы

- PascalCase.
- Имя — существительное.
- Суффиксы используются только по принятому стандарту проекта.

---

## Интерфейсы

- PascalCase.
- Без префикса `I`.

---

## Методы

- camelCase.
- Имя начинается с глагола.
- Должно описывать выполняемое действие.

---

## Поля

- camelCase.
- Имя должно отражать содержимое.
- Булевы поля начинаются с `is`, `has` или `can`.

---

## Константы

- UPPER_SNAKE_CASE.
- Только `static final`.

---

## Enum

- Имя типа — PascalCase.
- Значения — UPPER_SNAKE_CASE.

---

## Тесты

- Имя тестового класса совпадает с тестируемым классом с суффиксом `Test`.
- Имя тестового метода должно описывать проверяемое поведение.

# 4. Структура пакетов

## Общие правила

- Структура пакетов должна соответствовать архитектуре проекта.
- Каждый пакет имеет одну ответственность.
- Глубина вложенности должна быть минимально необходимой.
- Запрещены пакеты без назначения.
- Запрещены циклические зависимости между пакетами.

---

## Организация

- Структура строится по функциональным областям, а не по техническим слоям.
- Внутренние пакеты должны отражать назначение компонентов.
- Общие компоненты размещаются только в Platform Core.
- Запрещено дублирование одинаковых структур в разных модулях.

---

## Доступность

- Используется максимально возможный уровень инкапсуляции.
- Публичными являются только элементы, входящие в API модуля.
- Остальные классы должны иметь минимально необходимую область видимости.

---

## Зависимости

- Зависимости допускаются только в направлении, определённом архитектурой.
- Domain не зависит от Infrastructure.
- Domain не зависит от UI.
- UI взаимодействует только с Application API.
- Запрещены прямые обращения между независимыми Capability.

---

# 5. Правила Domain Layer

## Общие требования

- Domain является полностью независимым слоем.
- Domain не зависит от Spring.
- Domain не зависит от JavaFX.
- Domain не зависит от JPA.
- Domain не зависит от базы данных.
- Domain не зависит от внешних сервисов.
- Domain содержит только бизнес-логику.

---

## Модель предметной области

- Каждый Aggregate имеет один Aggregate Root.
- Entity обязаны иметь идентичность.
- Value Object должны быть неизменяемыми (immutable).
- Domain Service используется только при невозможности разместить логику в Aggregate.
- Инварианты должны проверяться внутри Domain.
- Нарушение бизнес-правил должно приводить к Domain Exception.
- Domain Event используется для публикации значимых бизнес-событий.

---

## Запрещается

- Использовать аннотации Spring.
- Использовать аннотации JPA.
- Использовать SQL.
- Использовать Repository напрямую.
- Выполнять HTTP-запросы.
- Выполнять файловые операции.
- Выполнять сетевые операции.
- Содержать код пользовательского интерфейса.

---

# 6. Правила Application Layer

## Общие требования

- Application реализует сценарии использования (Use Cases).
- Application координирует работу Domain.
- Application не содержит бизнес-правил.
- Application является единственной точкой входа в Domain.
- Все внешние взаимодействия проходят через Application.

---

## Организация

- Каждый Use Case реализуется отдельным Application Service.
- Входные данные принимаются через Command или Query.
- Результаты возвращаются через DTO.
- Все транзакции управляются только на уровне Application.
- Application использует Repository только через объявленные интерфейсы.

---

## Запрещается

- Содержать бизнес-логику.
- Выполнять прямые SQL-запросы.
- Использовать Entity Framework или JPA Entity в качестве DTO.
- Выполнять операции пользовательского интерфейса.
- Выполнять HTTP-вызовы напрямую, если они не инкапсулированы в Infrastructure Adapter.
- Нарушать границы Domain.

---

# 7. Правила Infrastructure Layer

## Общие требования

- Infrastructure реализует технические детали системы.
- Infrastructure зависит от Domain и Application.
- Infrastructure не содержит бизнес-логики.
- Все внешние интеграции реализуются только в Infrastructure.

---

## Организация

- Repository реализуют интерфейсы, объявленные в Domain.
- Persistence Adapter отвечает только за доступ к данным.
- External Adapter отвечает только за взаимодействие с внешними системами.
- Spring Configuration содержит только конфигурацию компонентов.
- Flyway используется исключительно для управления миграциями базы данных.

---

## Запрещается

- Содержать бизнес-правила.
- Изменять состояние Domain в обход Application.
- Использовать UI-компоненты.
- Передавать Persistence Entity за пределы Infrastructure.
- Использовать Repository напрямую из UI.

---

# 8. Правила UI Layer

## Общие требования

- UI отвечает только за взаимодействие с пользователем.
- UI не содержит бизнес-логики.
- UI взаимодействует только с Application API.
- UI не обращается напрямую к Domain.
- UI не обращается напрямую к Infrastructure.

---

## Организация

- FXML содержит только описание интерфейса.
- Controller обрабатывает события пользовательского интерфейса.
- ViewModel подготавливает данные для отображения.
- Долгие операции выполняются вне UI Thread.
- Навигация осуществляется через централизованный Navigation Service.

---

## Запрещается

- Размещать бизнес-логику в Controller.
- Выполнять SQL-запросы.
- Выполнять HTTP-запросы.
- Работать с Repository.
- Работать с Entity Domain напрямую.
- Блокировать UI Thread длительными операциями.

---

# 9. Правила тестирования

## Общие требования

- Все новые функции должны сопровождаться тестами.
- Каждый тест проверяет только один сценарий.
- Тесты должны быть независимыми.
- Тесты должны быть воспроизводимыми.
- Использование случайных данных допускается только при полном контроле результата.

---

## Unit Tests

- Проверяют бизнес-логику Domain.
- Не используют базу данных.
- Не используют Spring Context.
- Выполняются максимально быстро.

---

## Integration Tests

- Проверяют взаимодействие компонентов.
- Используют Testcontainers при работе с PostgreSQL.
- Проверяют Repository, миграции и конфигурацию.

---

## Architecture Tests

- Проверяют соблюдение архитектурных ограничений.
- Проверяют допустимые зависимости между модулями.
- Проверяют отсутствие запрещённых зависимостей.

---

## UI Tests

- Проверяют критические пользовательские сценарии.
- Не содержат бизнес-логики.

---

## Запрещается

- Игнорировать нестабильные тесты.
- Отключать тесты без согласования.
- Обходить неуспешные тесты ради завершения задачи.
- Объединять несколько независимых сценариев в одном тесте.

---

# 10. Правила Maven

## Общие требования

- Все модули входят в единый Maven Reactor.
- Все версии зависимостей централизованно управляются через `dependencyManagement`.
- Запрещено дублирование версий зависимостей в дочерних модулях.
- Каждый модуль должен собираться в составе общего проекта.

---

## Зависимости

- Используются только необходимые зависимости.
- Неиспользуемые зависимости запрещены.
- Версии библиотек должны быть согласованы между всеми модулями.
- Добавление новой зависимости требует архитектурного обоснования.

---

## Плагины

- Все плагины конфигурируются централизованно.
- Конфигурации не должны дублироваться между модулями.
- Используются только утверждённые плагины проекта.

---

## Проверки

Перед каждым Commit должны успешно выполняться:

- Maven Clean.
- Maven Compile.
- Maven Test.
- Maven Verify.

Сборка проекта без ошибок является обязательным условием завершения задачи.

---

# 11. Правила Git

## Общие требования

- Один Commit — одно логическое изменение.
- Один Commit относится только к одной задаче.
- История Commit должна быть понятной и последовательной.
- Все изменения должны быть отслеживаемыми.

---

## Commit

Перед созданием Commit обязательно:

- Проект успешно компилируется.
- Все тесты успешно проходят.
- Документация актуальна.
- Чек-листы актуальны.
- Журнал реализации обновлён.

---

## Запрещается

- Commit некомпилируемого кода.
- Commit с неуспешными тестами.
- Commit временного кода.
- Commit закомментированного кода.
- Commit несвязанных изменений.
- Прямое изменение файлов вне утверждённого Scope.

---

# 12. Definition of Done

Задача считается завершённой только при выполнении всех требований.

## Архитектура

- Соблюдены требования Architecture Overview.
- Соблюдены требования ADR.
- Не нарушены границы модулей.
- Не нарушены зависимости между слоями.

---

## Код

- Реализация завершена полностью.
- TODO, FIXME и HACK отсутствуют.
- Закомментированный код отсутствует.
- Неиспользуемый код отсутствует.
- Предупреждения компилятора отсутствуют.

---

## Качество

- Code Quality Standards соблюдены.
- Код отформатирован.
- Статический анализ выполнен успешно.

---

## Тестирование

- Все обязательные тесты реализованы.
- Все тесты успешно проходят.
- Maven Verify выполнен успешно.

---

## Документация

При необходимости обновлены:

- Specifications.
- ADR.
- Implementation Plan.
- Implementation Checklist.
- Implementation Log.

---

## Завершение

Перед закрытием задачи Cursor AI обязан подтвердить:

- Scope реализован полностью.
- Критерии завершения выполнены.
- Документация синхронизирована с кодом.
- Чек-листы обновлены.
- Реализация готова к Code Review.

После выполнения всех требований задача получает статус:

**Done**
