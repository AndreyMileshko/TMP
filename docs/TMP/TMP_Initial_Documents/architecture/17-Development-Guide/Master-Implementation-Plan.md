# Master Implementation Plan

**Document ID:** TMP-PLAN-001  
**Status:** Accepted  
**Version:** 1.0

---

# Назначение

Master Implementation Plan является главным документом реализации TOP Manufacturing Platform.

Документ определяет:

- полный порядок реализации проекта;
- последовательность этапов разработки;
- зависимости между этапами;
- правила выполнения работ;
- критерии начала и завершения каждого этапа;
- обязательные артефакты каждого этапа.

Данный документ является единственным источником порядка реализации проекта.

Все планы более низкого уровня должны соответствовать Master Implementation Plan.

---

# Область действия

Документ обязателен для:

- разработчиков;
- архитекторов;
- Cursor AI;
- других AI-агентов проекта.

Любая реализация, противоречащая настоящему документу, считается нарушением архитектуры проекта.

---

# Цели документа

Master Implementation Plan должен обеспечить:

- последовательную реализацию проекта;
- минимальный объём рабочего контекста AI;
- независимость этапов разработки;
- возможность безопасного продолжения проекта после любого перерыва;
- полную воспроизводимость процесса разработки;
- единый порядок выполнения задач.

---

# Основные принципы реализации

Разработка проекта выполняется исключительно поэтапно.

Каждый этап представляет собой полностью законченный логический блок.

Переход к следующему этапу допускается только после полного завершения предыдущего.

Запрещается:

- выполнять несколько этапов одновременно;
- изменять Scope завершённого этапа без архитектурного решения;
- пропускать этапы реализации;
- выполнять задачи вне утверждённого плана.

---

# Иерархия планирования

Для управления разработкой используется следующая структура документов.

```
Master Implementation Plan
        │
        ├── Stage Plan
        │       │
        │       ├── Implementation Checklist
        │       ├── Cursor Prompt
        │       ├── Implementation Log
        │       └── Stage Documentation
        │
        └── Следующий Stage
```

Каждый уровень документации имеет собственную область ответственности.

Master Implementation Plan определяет последовательность реализации.

Stage Plan определяет реализацию конкретного этапа.

Implementation Checklist определяет контроль выполнения.

Cursor Prompt ограничивает контекст AI.

Implementation Log фиксирует фактически выполненные изменения.

Ни один документ нижнего уровня не может изменять решения документов более высокого уровня.

---

# Правила использования плана

Master Implementation Plan используется только совместно с документами конкретного этапа.

Для каждого этапа обязательно создаются:

- Stage Plan;
- Implementation Checklist;
- Cursor Prompts;
- Implementation Log.

Работа начинается только после утверждения всех обязательных документов этапа.

Запрещается начинать реализацию по неполному комплекту документов.

---

# Правила декомпозиции задач

Каждый этап должен быть разделён на небольшие независимые задачи.

Каждая задача должна:

- иметь один измеримый результат;
- изменять минимально необходимое количество файлов;
- иметь точный Scope;
- иметь критерии начала;
- иметь критерии завершения;
- иметь перечень обязательных проверок;
- помещаться в один рабочий контекст Cursor AI.

Одна задача не должна одновременно включать:

- изменение архитектуры;
- реализацию нескольких модулей;
- рефакторинг соседнего функционала;
- исправление несвязанных ошибок;
- обновление файлов вне утверждённого Scope.

Если задача не может быть проверена отдельно, она должна быть декомпозирована.

---

# Размер задачи

Предпочтительный размер задачи:

- один компонент;
- один Use Case;
- один адаптер;
- одна миграция;
- одна View или ViewModel;
- один набор связанных тестов.

Допускается объединение нескольких изменений только при невозможности получить работоспособный результат по отдельности.

Причина объединения должна быть зафиксирована в Stage Plan.

---

# Scope задачи

Перед началом задачи обязательно фиксируются:

- цель;
- входные данные;
- разрешённые файлы;
- запрещённые изменения;
- ожидаемый результат;
- критерии приёмки;
- обязательные проверки.

Cursor AI запрещено изменять файлы, не указанные в Scope.

При обнаружении необходимости изменить дополнительный файл работа должна быть остановлена.

Изменение Scope допускается только после отдельного согласования.

---

# Question Gate

Перед написанием кода Cursor AI обязан проверить полноту задачи.

Cursor AI обязан задать вопросы до начала реализации, если:

- требования допускают несколько трактовок;
- отсутствует необходимое архитектурное решение;
- не определена модель данных;
- не определены зависимости;
- Scope противоречит документации;
- реализация требует изменения утверждённой архитектуры;
- отсутствуют критерии приёмки;
- невозможно определить ожидаемое поведение.

До получения ответов писать код запрещено.

Если блокирующих вопросов нет, Cursor AI обязан явно подтвердить:

**Question Gate passed. Blocking questions: none.**

---

# Правила управления контекстом Cursor AI

Каждая задача выполняется в отдельном чате Cursor AI.

Контекст чата должен содержать только информацию, необходимую для текущей задачи.

Запрещается автоматически добавлять в контекст:

- весь репозиторий;
- все спецификации проекта;
- документы завершённых этапов;
- код несвязанных модулей;
- полную историю реализации;
- соседние Stage Plan;
- файлы вне Scope.

---

# Обязательный контекст задачи

В контекст задачи передаются только:

- Cursor AI Guide;
- относящиеся к задаче разделы Architecture Overview;
- относящиеся ADR;
- Code Quality Standards;
- Stage Plan;
- актуальный фрагмент Implementation Checklist;
- точный Cursor Prompt;
- файлы, указанные в Scope;
- зависимости, необходимые для понимания контрактов.

Полные документы передаются только при невозможности выделить необходимый фрагмент.

---

# Порядок предоставления контекста

Контекст передаётся в следующем порядке:

1. Роль Cursor AI.
2. Цель задачи.
3. Обязательные правила.
4. Архитектурные ограничения.
5. Scope.
6. Исходные файлы.
7. Требуемый результат.
8. Критерии приёмки.
9. Обязательные проверки.
10. Формат отчёта.

---

# Контекст зависимостей

Если задача зависит от другого модуля, в контекст передаются только:

- публичный Application API;
- публичные Domain contracts;
- события;
- DTO;
- необходимые интерфейсы;
- краткое описание зависимости.

Внутренняя реализация зависимого модуля не передаётся, если она не требуется для выполнения задачи.

---

# Переполнение контекста

Если объём контекста становится слишком большим, задача должна быть остановлена и разделена.

Запрещается компенсировать переполнение контекста:

- сокращением обязательных требований;
- исключением критериев приёмки;
- передачей неполных архитектурных ограничений;
- разрешением Cursor AI самостоятельно изучать весь проект.

---

# Восстановление контекста

При продолжении работы в новом чате Cursor AI должен получить:

- цель текущей задачи;
- текущее состояние реализации;
- выполненные пункты Checklist;
- незавершённые пункты;
- актуальный Scope;
- изменённые файлы;
- результаты последних проверок;
- открытые вопросы.

История предыдущего чата не является обязательным источником контекста.

Источником текущего состояния является Implementation Log.

---

# Завершение задачи Cursor AI

После выполнения задачи Cursor AI обязан предоставить отчёт:

- что реализовано;
- какие файлы изменены;
- какие тесты добавлены;
- какие проверки выполнены;
- результаты проверок;
- какие документы обновлены;
- какие пункты Checklist отмечены;
- какие вопросы остались открыты.

Фраза «задача выполнена» без отчёта не допускается.

---

# Запрет самостоятельного расширения задачи

Cursor AI запрещено:

- реализовывать следующий пункт Checklist;
- исправлять соседние проблемы;
- выполнять необязательный рефакторинг;
- добавлять новые зависимости;
- менять публичные контракты;
- менять архитектуру;
- создавать дополнительные документы;
- изменять структуру модулей.

Любое необходимое отклонение должно быть сначала согласовано.

---

Реализация TOP Manufacturing Platform выполняется строго последовательно.

Каждый этап имеет:

- фиксированный Scope;
- собственный комплект документов;
- собственный Stage Plan;
- собственный Implementation Checklist;
- собственный Implementation Log;
- собственные Cursor Prompts;
- собственные критерии завершения.

Переход к следующему этапу допускается только после полного завершения текущего.

Запрещается выполнять работы следующего этапа до официального закрытия предыдущего.

# Stage 0 — Development Foundation

## Назначение

Development Foundation создаёт технический фундамент проекта.

После завершения Stage 0 проект должен успешно запускаться, собираться и быть полностью готовым для реализации бизнес-функциональности.

На данном этапе запрещается реализация предметной логики.

---

## Цели

Stage 0 должен обеспечить:

- единую структуру проекта;
- полностью настроенную инфраструктуру разработки;
- возможность локальной сборки;
- возможность автоматического тестирования;
- запуск Desktop Application;
- готовность Platform Core к реализации.

---

## Scope Stage 0

Разрешается:

- создание Maven Reactor;
- создание модулей проекта;
- настройка зависимостей;
- настройка Spring Boot;
- настройка JavaFX;
- настройка Bootstrap;
- настройка PostgreSQL;
- настройка Flyway;
- настройка Testcontainers;
- настройка jlink;
- настройка jpackage;
- настройка CI-проверок;
- создание минимальной структуры каталогов.

Запрещается:

- писать бизнес-логику;
- реализовывать Capability;
- реализовывать пользователей;
- реализовывать документы;
- реализовывать события предметной области;
- создавать рабочие экраны приложения.

---

## Результат Stage 0

По окончании этапа должны существовать:

- полностью собираемый Maven Reactor;
- все базовые модули проекта;
- настроенный Spring Boot;
- настроенный JavaFX Bootstrap;
- подключённый PostgreSQL;
- работающий Flyway;
- работающие Testcontainers;
- работающий Desktop Launcher;
- работающая упаковка приложения;
- базовая структура проекта.

---

## Выходной результат

После запуска приложения разработчик должен увидеть:

- старт приложения;
- успешную инициализацию Spring Boot;
- успешную инициализацию JavaFX;
- подключение к PostgreSQL;
- применение миграций Flyway;
- отображение пустого главного окна приложения.

На данном этапе отсутствие бизнес-функциональности является нормальным поведением.

## Назначение

Stage 1 реализует платформенное ядро системы.

Все последующие этапы используют Platform Core как единственную точку доступа к инфраструктурным сервисам.

---

## Scope

Разрешается реализация:

- Platform Registry;
- Event Bus;
- Service Registry;
- Capability Registry;
- Lifecycle Management;
- Platform Configuration;
- Core API.

Запрещается реализация любых бизнес-модулей.

---

## Результат

После завершения этапа платформа должна:

- успешно регистрировать сервисы;
- регистрировать Capability;
- публиковать события;
- принимать события;
- корректно завершать жизненный цикл компонентов.

---

# Stage 2 — Document Engine

## Назначение

Реализовать платформенную подсистему управления документами.

---

## Scope

- Document Registry;

## Результат

---

---

# Обязательные документы каждого этапа

Каждый Stage должен иметь собственный комплект документов.

Начало реализации этапа без полного комплекта документов запрещается.

Минимальный обязательный комплект включает:

| Документ | Назначение |
|----------|------------|
| Stage Plan | План реализации этапа |
| Implementation Checklist | Контроль выполнения |
| Cursor Prompts | Промпты для Cursor AI |
| Implementation Log | Журнал реализации |
| Stage ADR (при необходимости) | Архитектурные изменения |
| Updated Documentation | Актуализированная документация |

При необходимости допускается создание дополнительных технических документов, если это предусмотрено Stage Plan.

---

# Порядок выполнения этапа

Каждый Stage выполняется по единому алгоритму.

## Шаг 1. Подготовка

Перед началом реализации необходимо:

- утвердить Stage Plan;
- утвердить Checklist;
- подготовить Cursor Prompt;
- определить Scope;
- определить критерии завершения;
- проверить зависимости.

---

## Шаг 2. Декомпозиция

Stage разбивается на независимые задачи.

Каждая задача должна:

- иметь собственный Scope;
- иметь собственные критерии приёмки;
- иметь собственный набор файлов;
- иметь собственные проверки.

Запрещается создавать задачи без измеримого результата.

---

## Шаг 3. Реализация

Для каждой задачи Cursor AI обязан:

- пройти Question Gate;
- выполнить только утверждённый Scope;
- соблюдать Code Quality Standards;
- соблюдать Architecture Overview;
- соблюдать ADR;
- соблюдать ограничения Stage Plan.

---

## Шаг 4. Проверка

После реализации обязательно выполняются:

- компиляция;
- Unit Tests;
- Integration Tests (если предусмотрены);
- Architecture Tests;
- статический анализ;
- проверка форматирования;
- проверка зависимостей;
- проверка миграций (при наличии);
- проверка JavaFX (если изменён UI).

При наличии хотя бы одной ошибки задача считается незавершённой.

---

## Шаг 5. Документирование

После успешной проверки Cursor AI обязан:

- обновить Checklist;
- обновить Implementation Log;
- обновить техническую документацию;
- обновить комментарии в Stage Plan при необходимости.

---

## Шаг 6. Закрытие задачи

Задача считается завершённой только после выполнения всех условий:

- код реализован;
- проверки выполнены успешно;
- документация обновлена;
- Checklist обновлён;
- Implementation Log обновлён;
- отчёт Cursor AI сформирован.

---

# Контрольные точки

Для контроля выполнения используются следующие контрольные точки.

## Gate 0 — Start Ready

Проверяется:

- утверждён Scope;
- отсутствуют открытые архитектурные вопросы;
- подготовлен Prompt;
- подготовлен Checklist.

Без прохождения Gate 0 разработка запрещена.

---

## Gate 1 — Code Complete

Проверяется:

- реализован весь Scope;
- отсутствуют TODO;
- отсутствует временный код;
- отсутствуют заглушки;
- отсутствует отладочный код.

---

## Gate 2 — Verification Complete

Проверяется:

- проходят все проверки;
- отсутствуют ошибки сборки;
- отсутствуют предупреждения, запрещённые стандартами проекта;
- выполнены архитектурные проверки.

---

## Gate 3 — Documentation Complete

Проверяется:

- обновлена документация;

# Порядок перехода между этапами

Переход к следующему Stage выполняется только после официального закрытия предыдущего.

Перед началом нового Stage необходимо:

1. Подтвердить выполнение всех критериев завершения предыдущего этапа.
2. Закрыть все пункты соответствующего Implementation Checklist.
3. Убедиться в отсутствии блокирующих дефектов.
4. Утвердить Stage Plan следующего этапа.
5. Подготовить новый комплект Cursor Prompts.
6. Подготовить новый Implementation Checklist.
7. Подготовить новый Implementation Log.
8. Проверить актуальность архитектурной документации.

После этого предыдущий Stage получает статус **Completed**, а следующий — **In Progress**.

Одновременное выполнение нескольких Stage запрещено.
