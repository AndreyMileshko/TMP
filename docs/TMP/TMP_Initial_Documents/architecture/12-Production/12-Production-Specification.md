# Production Specification

**Document ID:** TMP-SPEC-012  
**Status:** Accepted  
**Version:** 1.1

---

# 1. Назначение

Production — функциональная область TOP Manufacturing Platform (TMP), отвечающая за производственное состояние позиций заказа, формирование производственных партий, запуск позиций в производство и фиксацию выпуска изделий.

Production работает с позицией заказа как с главным производственным объектом платформы и не создаёт отдельную бизнес-сущность Production Order.

Производственная партия является частью документов Production и определяет конкретное количество изделий одной или нескольких позиций заказа, запускаемое или выпускаемое в рамках бизнес-операции.

Production не описывает внутреннее выполнение технологических операций и не является MES-системой.

---

# 2. Область ответственности

Production отвечает за:

- хранение производственного статуса позиции заказа;
- хранение количества изделий, запущенного в производство;
- хранение накопленного количества выпуска по позиции заказа;
- проверку обеспеченности позиций заказа материалами;
- получение утверждённых карт раскроя для конкретных производственных партий;
- формирование рекомендаций для перемещения материалов на склад производства;
- запуск производственных партий;
- отмену ранее выполненного запуска;
- фиксацию полного или частичного выпуска изделий;
- передачу Warehouse данных для резервирования и списания материалов;
- публикацию событий начала и завершения использования производственной партии;
- предоставление производственного состояния через Public API;
- аудит производственных действий и изменений статусов.

Production не отвечает за:

- создание и изменение заказов клиентов;
- владение позициями заказа;
- создание и изменение спецификаций;
- создание и изменение карт раскроя;
- автоматическую оптимизацию раскроя;
- хранение мастер-данных материалов;
- складские остатки и складские движения;
- непосредственное резервирование или списание материалов;
- создание и проведение складских документов;
- технологические маршруты;
- прохождение технологических этапов;
- регистрацию фактически выполненных технологических операций;
- производственные участки и рабочие центры;
- управление браком, переделкой и остановкой производства;
- календарное и оперативное планирование;
- расчёт загрузки оборудования;
- контроль качества;
- склад готовой продукции;
- отгрузку, монтаж и сервис;
- производственную аналитику, отчётность и KPI.

> **Architecture Rule**  
> Production изменяет только собственное производственное состояние и собственные документы и не изменяет данные других Capability напрямую.

---

# 3. Основные принципы

1. Главным производственным объектом остаётся позиция заказа.
2. Production не создаёт отдельный объект Production Order.
3. Производственный запуск и выпуск выполняются в отношении конкретного количества изделий.
4. Количество одной позиции заказа может быть разделено между несколькими производственными партиями.
5. Все значимые изменения производственного состояния выполняются посредством бизнес-документов.
6. Проверка готовности является пользовательской командой, а не бизнес-документом.
7. Спецификация позиции заказа является единственным источником истины о составе изделия.
8. Production не хранит копию спецификации.
9. Утверждённая спецификация позиции заказа является неизменяемой.
10. Для длинномерного материала утверждённый Cutting Plan является источником потребности в исходном материале для включённой в него производственной партии.
11. Cutting Plan не изменяет спецификацию изделия.
12. При отсутствии утверждённого Cutting Plan потребность определяется по спецификации.
13. Production использует только утверждённую Revision Cutting Plan.
14. Production не создаёт, не изменяет и не пересчитывает Cutting Plan.
15. Warehouse является единственным владельцем остатков, резервов и складских движений.
16. Выпуск изделия фиксируется только после успешного списания материалов со склада производства.
17. Склад готовой продукции не входит в версию 1.0.
18. Частичный запуск и частичный выпуск разрешены.
19. Производственный статус заказа целиком не хранится в Production.
20. Аналитика не входит в Public API Production.
21. Проведённые документы Production не изменяются.
22. Исправление ранее выполненного запуска производится новым компенсирующим документом.
23. Production публикует бизнес-события, необходимые Cutting Optimization для управления жизненным циклом карты раскроя.

---

# 4. Владение данными

Production является владельцем:

- производственного статуса позиции заказа;
- количества изделий, запущенного в производство;
- количества изделий, находящегося в активном производстве;
- накопленного количества выпуска по позиции заказа;
- производственных партий в составе документов Production;
- документов Production;
- результатов обработки строк документов Production;
- ссылок на применённые Revision Cutting Plan;
- настроек Production;
- истории изменения собственного производственного состояния.

Production не владеет:

- заказом клиента;
- позицией заказа как бизнес-объектом;
- Revision позиции заказа;
- спецификацией позиции заказа;
- материалами;
- Cutting Plan;
- Source Bar;
- Cut Piece;
- Batch Allocation Cutting Optimization;
- складами и ячейками;
- Stock Position;
- Warehouse Movement;
- Warehouse Operation;
- резервами;
- аналитическими проекциями.

Доступ к данным других функциональных областей выполняется только через их публичные контракты и Domain Events.

# 5. Основной объект Production

Основным объектом Production является **производственное состояние Revision позиции заказа**.

Production связывает собственное состояние с:

- идентификатором позиции заказа;
- конкретной Revision позиции заказа.

Владельцем позиции и её Revision остаётся Order Management.

## 5.1 Хранимые данные

Для Revision позиции заказа Production хранит как минимум:

| Атрибут | Описание |
|----------|----------|
| Order Item ID | Идентификатор позиции заказа |
| Order Item Revision | Revision позиции заказа |
| Production Status | Текущий производственный статус |
| Ordered Quantity | Количество изделий Revision позиции заказа |
| Launched Quantity | Суммарное количество изделий, запуск которых зафиксирован |
| Active Production Quantity | Количество изделий, находящееся в активном производстве |
| Released Quantity | Суммарно выпущенное количество |
| Remaining Quantity | Количество, оставшееся к выпуску |
| Available To Launch Quantity | Количество, доступное для нового запуска |
| Last Readiness Check At | Дата и время последней проверки готовности |
| Last Status Changed At | Дата и время последнего изменения статуса |

`Remaining Quantity` вычисляется как:

```text
Remaining Quantity =
Ordered Quantity
−
Released Quantity
```

`Available To Launch Quantity` вычисляется как:

```text
Available To Launch Quantity =
Ordered Quantity
−
Released Quantity
−
Active Production Quantity
```

Вычисляемые количества не являются самостоятельными источниками истины.

> **Architecture Rule**  
> Production не дублирует коммерческие, конструктивные, технологические и складские данные позиции заказа.

---

# 6. Производственная партия

Производственная партия представляет конкретное количество изделий, включённое в документ Production.

Производственная партия не является самостоятельным бизнес-документом.

Она определяется строками документа Production.

## 6.1 Состав производственной партии

Для каждой строки производственной партии указываются:

- Order ID;
- Order Item ID;
- Order Item Revision;
- количество изделий;
- применяемая Revision Cutting Plan (если существует);
- источник потребности в материалах;
- документ-основание.

---

## 6.2 Разделение позиции заказа

Количество одной Revision позиции заказа может быть разделено между несколькими производственными партиями.

Например:

```text
Order Item №125 Rev.2

100 изделий

↓

Production Launch №1
40 изделий

↓

Production Launch №2
35 изделий

↓

Production Launch №3
25 изделий
```

Суммарное количество изделий в активном производстве и выпущенное количество не могут превышать количество изделий соответствующей Revision позиции заказа.

---

## 6.3 Связь с Cutting Plan

Если для производственной партии существует утверждённая Revision Cutting Plan:

- Production использует только данную Revision;
- количество запуска не может превышать количество изделий, включённое в Batch Allocation;
- ссылка на Revision Cutting Plan сохраняется в документе Production;
- после проведения документа ссылка на карту раскроя не изменяется.

Production никогда:

- не создаёт Cutting Plan;
- не изменяет Cutting Plan;
- не пересчитывает Cutting Plan;
- не выбирает другую Revision Cutting Plan автоматически.

Если одна карта раскроя охватывает несколько заказов, для каждого заказа могут существовать отдельные документы Production, ссылающиеся на одну и ту же Revision Cutting Plan.

---

# 7. Производственные статусы позиции заказа

Production поддерживает следующие статусы.

| Код | Назначение |
|------|------------|
| `NOT_STARTED` | Производство не начиналось |
| `READY_FOR_PRODUCTION` | Материалов достаточно для запуска |
| `IN_PRODUCTION` | Выполняется производство |
| `PARTIALLY_RELEASED` | Выпущена часть изделий |
| `RELEASED` | Выпущено всё количество изделий |
| `CANCELLED` | Производство отменено |

---

## 7.1 Переходы статусов

```text
NOT_STARTED
      │
      ▼
READY_FOR_PRODUCTION
      │
      ▼
IN_PRODUCTION
      │
      ├───────────────┐
      ▼               │
PARTIALLY_RELEASED    │
      │               │
      ▼               │
RELEASED              │
                      │
NOT_STARTED ─────────► CANCELLED
READY_FOR_PRODUCTION ─► CANCELLED
IN_PRODUCTION ───────► CANCELLED
```

Статус определяется агрегированно по количествам соответствующей Revision позиции заказа.

Переход в:

- `PARTIALLY_RELEASED`;
- `RELEASED`;

выполняется только документом **«Выпуск изделий»**.

Уменьшение количества изделий в активном производстве выполняется только документами:

- «Выпуск изделий»;
- «Отмена запуска в производство».

---

# 8. Проверка готовности заказа к производству

Проверка готовности является пользовательской командой:

```text
Проверить готовность заказа к производству
```

Команда не создаёт бизнес-документ.

---

## 8.1 Ответственность Production

Production:

1. получает заказ и Revision позиций из Order Management;
2. определяет количество изделий, оставшееся к выпуску;
3. получает спецификации Revision позиций;
4. получает утверждённые Revision Cutting Plan;
5. формирует потребность в материалах;
6. запрашивает Warehouse о наличии материалов;
7. определяет готовность каждой Revision позиции;
8. устанавливает производственный статус;
9. возвращает пользователю результат проверки.

Warehouse предоставляет сведения о наличии материалов.

Cutting Optimization предоставляет технологические данные утверждённых карт раскроя.

Ни один из этих Capability не изменяет производственное состояние.

---

## 8.2 Источник потребности

Для каждого материала используется только один источник определения потребности.

### Длинномерные материалы

Если существует утверждённая Revision Cutting Plan, включающая соответствующую производственную партию, потребность определяется по Cutting Plan.

Используются:

- Material ID;
- артикул;
- цвет;
- длины исходных хлыстов;
- количество исходных хлыстов;
- Revision Cutting Plan.

Production не использует внутренние объекты Cutting Optimization:

- Cut Piece;
- Source Bar;
- Batch Allocation.

Production использует только итоговую технологическую потребность, рассчитанную Capability Cutting Optimization.

---

### Остальные материалы

Если утверждённый Cutting Plan отсутствует либо материал не входит в карту раскроя, потребность определяется по спецификации Revision позиции заказа.

Расчёт выполняется пропорционально количеству изделий производственной партии.

---

### Смешанный сценарий

Для одной производственной партии одновременно допускается использование:

- Cutting Plan — для длинномерных материалов;
- Specification — для остальных материалов.

Один и тот же материал не должен одновременно учитываться по двум источникам.

---

## 8.3 Используемые складские остатки

При проверке готовности учитываются материалы:

- на основном складе;
- на складе производства.

Warehouse самостоятельно определяет, какие остатки являются доступными для обеспечения производства.

Production не содержит собственной логики расчёта складских остатков.

---

## 8.4 Результат проверки

Если материалов достаточно:

```text
NOT_STARTED
        │
        ▼
READY_FOR_PRODUCTION
```

Если материалов недостаточно:

```text
READY_FOR_PRODUCTION
        │
        ▼
NOT_STARTED
```

Пользователю отображаются:

- материал;
- источник потребности;
- применяемая Revision Cutting Plan (если используется);
- требуемое количество;
- количество на основном складе;
- количество на складе производства;
- суммарное доступное количество;
- недостающее количество;
- итоговый результат проверки.

Результат определяется отдельно для каждой Revision позиции заказа.

---

# 9. Рекомендация для перемещения материалов

Production предоставляет команду формирования рекомендации по перемещению материалов на склад производства.

Рекомендация может формироваться:

- по одному заказу;
- по нескольким заказам;
- по выбранным Revision позиций;
- по одной или нескольким производственным партиям.

Production определяет:

> Какие материалы рекомендуется переместить для обеспечения выбранных производственных партий.

Warehouse определяет:

> Каким образом выполнить складское перемещение.

Production никогда самостоятельно не создаёт складской документ.

---

## 9.1 Результат формирования рекомендации

Production формирует агрегированный перечень рекомендуемых материалов.

Для каждого материала сохраняется детализация происхождения потребности:

- Order ID;
- Order Item ID;
- Revision позиции;
- производственная партия;
- количество изделий;
- Revision Cutting Plan (если используется);
- источник потребности;
- рекомендуемое количество материала.

После передачи рекомендации владельцем складского документа становится Warehouse.

---

## 9.2 Приоритет Cutting Plan

Для длинномерных материалов применяются следующие правила.

### Если существует утверждённый Cutting Plan

Количество исходного материала определяется исключительно по утверждённой Revision Cutting Plan.

Используется:

- итоговое количество исходных хлыстов;
- длины хлыстов;
- материал;
- Revision Cutting Plan.

Спецификация изделия для расчёта длинномерного материала не используется.

---

### Если Cutting Plan отсутствует

Количество определяется по спецификации Revision позиции заказа.

Расчёт выполняется пропорционально количеству изделий производственной партии.

---

## 9.3 Смешанный расчёт

В рамках одной производственной партии допускается одновременное использование:

- Cutting Plan — для длинномерных материалов;
- Specification — для остальных материалов.

Для одного материала допускается только один источник расчёта.

---

## 9.4 Хранение рекомендации

Рекомендация не является самостоятельным бизнес-документом Production.

После передачи данных Warehouse самостоятельно:

- создаёт черновик складского документа;
- выполняет проверку;
- проводит документ;
- регистрирует складские движения.

Production не управляет жизненным циклом складских документов.

---

# 10. Документы Production

Production регистрирует следующие типы документов:

- Production Launch;
- Production Launch Cancellation;
- Production Release.

Каждому типу документа соответствует собственный Document Processor.

После проведения документ становится неизменяемым.

Исправление выполняется исключительно новым документом.

Все документы Production сохраняют ссылки на:

- Revision позиции заказа;
- применяемую Revision Cutting Plan (если используется);
- документ-основание;
- производственную партию.

Проведённый документ никогда не изменяется.

---

# 11. Запуск в производство

Документ **Production Launch** фиксирует начало производства конкретной производственной партии.

Документ не создаёт производственную партию, а использует уже определённую пользователем партию.

---

## 11.1 Состав документа

Каждая строка документа содержит:

- Order ID;
- Order Item ID;
- Order Item Revision;
- количество изделий;
- ссылку на применяемую Revision Cutting Plan (если используется);
- источник потребности в материалах;
- статус обработки строки.

---

## 11.2 Проверки перед проведением

Перед проведением документа Production выполняет проверку:

- существования Revision позиции заказа;
- корректности количества изделий;
- отсутствия превышения доступного количества к запуску;
- существования применяемой Revision Cutting Plan (если используется);
- принадлежности производственной партии указанной Revision Cutting Plan;
- отсутствия отменённой или архивной Revision Cutting Plan;
- готовности материалов по данным Warehouse.

При наличии ошибок проведение документа запрещается.

---

## 11.3 Проведение документа

После успешного проведения документа:

- увеличивается `Launched Quantity`;
- увеличивается `Active Production Quantity`;
- пересчитывается производственный статус;
- публикуется Domain Event:

```text
ProductionLaunchStarted
```

Если документ использует Cutting Plan, событие приводит к переводу соответствующей карты раскроя в состояние `IN_USE`.

---

# 12. Отмена запуска

Документ **Production Launch Cancellation** используется для отмены ранее выполненного запуска.

Документ не изменяет ранее проведённый Production Launch.

---

## 12.1 Проведение документа

После проведения:

- уменьшается `Launched Quantity`;
- уменьшается `Active Production Quantity`;
- пересчитывается производственный статус;
- публикуются необходимые внутренние события Production.

Если после отмены активное количество становится равным нулю, статус определяется повторно по общим правилам.

---

# 13. Выпуск изделий

Документ **Production Release** фиксирует выпуск готовой продукции.

---

## 13.1 Проверки

Перед проведением проверяется:

- наличие активного количества в производстве;
- корректность количества выпуска;
- отсутствие превышения количества изделий в производстве;
- успешное списание материалов Warehouse.

Если списание материалов не выполнено, проведение документа запрещается.

---

## 13.2 Проведение документа

После проведения:

- уменьшается `Active Production Quantity`;
- увеличивается `Released Quantity`;
- пересчитывается `Remaining Quantity`;
- пересчитывается производственный статус.

Если производственная партия полностью завершена, публикуется событие:

```text
ProductionLaunchCompleted
```

Capability Cutting Optimization самостоятельно обрабатывает данное событие, переводит соответствующую Revision Cutting Plan в статус `COMPLETED` и публикует Domain Event `CuttingPlanCompleted`.

---

# 14. Public API

Production предоставляет следующий Public API.

```text
checkProductionReadiness(orderId)
```

Проверка готовности заказа к производству.

---

```text
createProductionLaunch(request)
```

Создание документа запуска в производство.

---

```text
cancelProductionLaunch(documentId)
```

Создание документа отмены запуска.

---

```text
createProductionRelease(request)
```

Создание документа выпуска изделий.

---

```text
getProductionState(orderItemId, revision)
```

Получение текущего производственного состояния Revision позиции заказа.

---

```text
getProductionReadiness(orderId)
```

Получение результата последней проверки готовности.

---

```text
generateMaterialTransferRecommendation(request)
```

Формирование рекомендации для перемещения материалов на склад производства.

Production возвращает перечень рекомендуемых материалов.

Создание складского документа выполняется Capability Warehouse.

---

# 15. Domain Events

Production публикует только завершённые бизнес-события.

---

## ProductionLaunchStarted

Публикуется после проведения документа Production Launch.

Используется Cutting Optimization для перевода соответствующей Revision Cutting Plan в статус `IN_USE`.

---

## ProductionLaunchCompleted

Публикуется после полного завершения производственной партии.

Используется Cutting Optimization для перевода соответствующей Revision Cutting Plan в статус `COMPLETED`.

---

## ProductionReleased

Публикуется после успешного выпуска изделий.

Используется другими Capability при необходимости.

---

## ProductionLaunchCancelled

Публикуется после проведения документа отмены запуска.

Используется внутренними подписчиками платформы.

---

# 16. Получение Domain Events

Production подписывается на Domain Events других Capability.

В версии 1.1 используются:

- Order Item Revision Approved;
- CuttingPlanApproved;
- WarehouseDocumentPosted;
- WarehouseDocumentCancelled.

Production самостоятельно принимает решение об изменении собственного состояния на основании полученных событий.

Ни один внешний Capability не изменяет документы Production напрямую.

---

# 17. Security

Production регистрирует следующие Capability в Security.

| Capability | Назначение |
|------------|------------|
| `PRODUCTION_VIEW` | Просмотр производственного состояния |
| `PRODUCTION_CHECK_READINESS` | Проверка готовности заказа к производству |
| `PRODUCTION_CREATE_LAUNCH` | Создание документа запуска в производство |
| `PRODUCTION_CANCEL_LAUNCH` | Создание документа отмены запуска |
| `PRODUCTION_CREATE_RELEASE` | Создание документа выпуска изделий |
| `PRODUCTION_VIEW_RECOMMENDATIONS` | Просмотр рекомендаций по перемещению материалов |

Security защищает только бизнес-операции.

Внутренние действия Production отдельными Capability не регистрируются.

---

# 18. Audit

Production ведёт аудит только значимых бизнес-событий.

Подлежат аудиту:

- выполнение проверки готовности;
- проведение документа Production Launch;
- проведение документа Production Launch Cancellation;
- проведение документа Production Release;
- изменение производственного статуса;
- публикация Domain Events.

Не подлежат аудиту:

- промежуточные вычисления;
- внутренние проверки;
- повторные расчёты количества;
- внутренние сервисные операции.

История изменений документов обеспечивается механизмом Document Engine.

---

# 19. Invariants

Следующие правила являются обязательными для Capability.

---

## Invariant 1

Production не изменяет данные Order Management.

---

## Invariant 2

Production не изменяет спецификацию изделия.

---

## Invariant 3

Production использует только утверждённую Revision позиции заказа.

---

## Invariant 4

Production использует только утверждённую Revision Cutting Plan.

---

## Invariant 5

При отсутствии утверждённой Revision Cutting Plan потребность определяется по спецификации.

---

## Invariant 6

Проведённые документы Production являются неизменяемыми.

---

## Invariant 7

Количество изделий, находящееся в активном производстве, не может быть отрицательным.

---

## Invariant 8

Суммарное количество выпущенных изделий не может превышать количество изделий Revision позиции заказа.

---

## Invariant 9

Количество изделий, запускаемое в производство, не может превышать доступное к запуску количество.

---

## Invariant 10

Production никогда не изменяет Cutting Plan.

---

## Invariant 11

Warehouse никогда не изменяет производственное состояние Production.

---

## Invariant 12

Все изменения собственного состояния Production выполняет самостоятельно на основании собственных документов или полученных Domain Events.

# 20. Architecture Rules

## Rule 1

Production является единственным владельцем собственного производственного состояния.

---

## Rule 2

Production никогда не изменяет документы других Capability.

---

## Rule 3

Все взаимодействие между Capability выполняется исключительно через Public API и Domain Events.

---

## Rule 4

Production использует Specification как единственный источник данных об изделии.

Cutting Plan используется исключительно как источник технологических данных для определения потребности в длинномерном материале.

---

## Rule 5

Warehouse является единственным владельцем:

- складских остатков;
- резервов;
- складских движений;
- складских документов.

---

## Rule 6

Cutting Optimization является единственным владельцем:

- Cutting Plan;
- Source Bar;
- Cut Piece;
- Batch Allocation.

Production использует только утверждённую Revision Cutting Plan.

---

## Rule 7

Order Management является единственным владельцем:

- Customer Order;
- Order Item;
- Revision позиции заказа;
- Specification.

---

## Rule 8

Все проведённые документы являются неизменяемыми.

Исправление выполняется созданием нового документа.

---

# 21. Version 1.0 Limitations

В версию 1.0 входят:

- запуск производственных партий;
- отмена запуска;
- выпуск изделий;
- частичный запуск;
- частичный выпуск;
- проверка готовности;
- интеграция с Warehouse;
- интеграция с Order Management;
- интеграция с Cutting Optimization;
- использование утверждённых карт раскроя;
- рекомендации по перемещению материалов.

В версию 1.0 не входят:

- MES;
- технологические маршруты;
- производственные операции;
- рабочие центры;
- оборудование;
- диспетчеризация производства;
- управление сменами;
- контроль выполнения операций;
- автоматическая регистрация выпуска с оборудования;
- OEE;
- производственная аналитика;
- календарное планирование;
- APS;
- управление складом готовой продукции.

# Итог

Production является платформенным Capability TOP Manufacturing Platform, отвечающим за производственное состояние позиций заказа, запуск и выпуск производственных партий.

Production использует:

- Specification — как источник информации об изделии;
- Cutting Plan — как источник технологических данных для длинномерных материалов;
- Warehouse — как владельца складских данных.

Production самостоятельно управляет собственными документами и производственным состоянием, не изменяя данные других Capability напрямую и взаимодействуя с ними исключительно посредством Public API и Domain Events.
