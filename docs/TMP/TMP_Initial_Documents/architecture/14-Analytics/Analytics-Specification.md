# Analytics Specification

**Document ID:** TMP-SPEC-014  
**Status:** Accepted  
**Version:** 1.0

---

# Назначение

Analytics является read-only Capability TOP Manufacturing Platform, предназначенным для простого и информативного анализа движения материалов.

В версии 1.0 Analytics предоставляет один основной отчёт — **«Движение материалов»**.

Отчёт позволяет просматривать:

- приходы материалов;
- расходы материалов;
- внутренние перемещения;
- резервирование и снятие резерва;
- списания на Customer Order и Order Item;
- остатки на начало и конец периода;
- складской оборот за выбранный период.

Analytics не изменяет складские документы, движения, остатки и другие бизнес-объекты.

---

# Основные принципы

1. Analytics является отдельным read-only Capability.
2. Источником истины являются Warehouse и проведённые складские документы.
3. Analytics не участвует в проведении документов.
4. Analytics не изменяет данные других Capability.
5. В MVP не создаётся собственное аналитическое хранилище.
6. Данные формируются по актуальному состоянию Warehouse.
7. Количества анализируются только в базовых единицах измерения материалов.
8. Денежная и финансовая аналитика в MVP отсутствует.
9. Разные единицы измерения не суммируются между собой.
10. Analytics не публикует Domain Events.

---

# Область MVP

В Analytics MVP входит:

- один отчёт «Движение материалов»;
- просмотр приходов, расходов и перемещений;
- просмотр резервирования и снятия резерва;
- расчёт остатков на начало и конец периода;
- фильтрация;
- сортировка;
- серверная пагинация;
- просмотр деталей движения;
- переход к связанным объектам при наличии прав;
- экспорт в XLSX;
- права просмотра и экспорта.

---

# Источники данных

Analytics получает данные через Public API Warehouse.

Используются:

- `Warehouse Movement` — неизменяемая история движений;
- `Stock Position` — текущие складские остатки;
- проведённые складские документы;
- ссылки на `Customer Order` и `Order Item`, если они присутствуют в складской операции.

Analytics не должен:

- обращаться к репозиториям других Capability;
- читать внутренние таблицы других Capability;
- использовать внутренние классы реализации других Capability;
- изменять складские документы, движения или остатки.

После проведения складского документа соответствующее движение должно отображаться после следующего обновления отчёта.

Синхронизация, фоновые витрины, постоянный кэш и отдельная аналитическая база в MVP не используются.

# Отчёт «Движение материалов»

## Назначение

Отчёт предназначен для просмотра всех изменений складского состояния материалов за выбранный период.

## Категории движения

В интерфейсе используются четыре категории:

1. **Приход**.
2. **Расход**.
3. **Перемещение**.
4. **Изменение состояния**.

Каждая строка дополнительно содержит исходный тип Warehouse Operation.

## Классификация операций

| Операция Warehouse | Категория Analytics | Влияние на общий остаток |
|---|---|---:|
| `Receipt` | Приход | Увеличивает |
| `Put Away` | Перемещение | Не изменяет |
| `Move` | Перемещение | Не изменяет |
| `Transfer` | Перемещение | Не изменяет общий остаток |
| `Receive Transfer` | Перемещение | Не изменяет общий остаток |
| `Consumption` | Расход | Уменьшает |
| `Return` | Перемещение | Не изменяет общий остаток |
| `Adjustment` с положительной дельтой | Приход | Увеличивает |
| `Adjustment` с отрицательной дельтой | Расход | Уменьшает |
| `Scrap` | Расход | Уменьшает |
| `Inventory` | Приход или расход | Зависит от выявленной разницы |
| `Reserve` | Изменение состояния | Не изменяет физический остаток |
| `Unreserve` | Изменение состояния | Не изменяет физический остаток |

Исходная Warehouse Operation остаётся источником истины для направления и количества движения.

## Правило межскладского перемещения

Межскладское перемещение:

- уменьшает остаток склада-источника;
- увеличивает остаток склада-получателя;
- не изменяет общий остаток по всем складам компании.

## Фильтры

Обязательные и доступные фильтры:

- период — обязательный;
- склад;
- материал;
- категория движения;
- Customer Order;
- Order Item.

Значения по умолчанию:

- период — текущий месяц;
- склад — все склады;
- материал — все материалы;
- категория — все движения.

Изменение фильтра не запускает запрос автоматически. Для применения используется кнопка `Применить`.

## Основная таблица

Таблица содержит:

| Поле | Назначение |
|---|---|
| Дата и время | Момент фактического складского движения |
| Категория | Приход, расход, перемещение или изменение состояния |
| Операция | Исходный тип Warehouse Operation |
| Код материала | Код материала |
| Материал | Наименование материала |
| Цвет | Цвет материала |
| Количество | Количество в базовой единице |
| Единица измерения | Базовая единица материала |
| Склад-источник | Склад списания или отправки |
| Склад-получатель | Склад поступления или получения |
| Документ | Тип и номер документа-основания |
| Customer Order | Связанный заказ, если существует |
| Order Item | Связанная позиция заказа, если существует |
| Пользователь | Пользователь, выполнивший операцию |

По умолчанию строки сортируются от новых к старым.

Допустима сортировка по:

- дате;
- материалу;
- складу;
- категории;
- количеству;
- заказу.

## Сводные показатели

Отчёт отображает:

- остаток на начало периода;
- приход за период;
- расход за период;
- внутреннее перемещение;
- остаток на конец периода;
- количество операций.

Показатели количества группируются по единицам измерения.

Пример:

```text
Приход:
- 1 250 м
- 8 400 шт.
- 36 кг
```

Суммирование разных единиц измерения запрещено.

## Детали движения

Двойное нажатие на строку открывает карточку движения в режиме чтения.

Карточка содержит:

- основные данные движения;
- склад и ячейку;
- материал и партию;
- количество и единицу измерения;
- исходное и новое состояние запаса;
- документ-основание;
- Customer Order;
- Order Item;
- пользователя;
- идентификатор Warehouse Operation;
- идентификатор Warehouse Movement;
- дату проведения документа.

# Правила расчёта

## Остаток на начало периода

Остаток на начало рассчитывается как результат всех действующих движений материала, проведённых до начала выбранного периода.

```text
Остаток на начало =
все приходы до периода
− все расходы до периода
```

Для отдельного склада учитываются только движения выбранного склада.

## Остаток на конец периода

```text
Остаток на конец =
остаток на начало
+ приход за период
− расход за период
```

## Резервирование

`Reserve` и `Unreserve`:

- отображаются в детализации;
- изменяют состояние запаса;
- не изменяют физическое количество материала;
- не включаются в приход и расход.

## Документы и корректировки

В отчёт не включаются:

- черновики;
- непроведённые документы;
- операции, не создавшие действующий Warehouse Movement.

Если исправление выполнено обратным движением, отображаются обе записи:

- исходная;
- корректирующая.

Исторические движения не удаляются и не переписываются.

# Пользовательский интерфейс

Раздел:

```text
Аналитика → Движение материалов
```

Экран состоит из:

1. панели фильтров;
2. сводных показателей;
3. таблицы движений;
4. панели действий.

Действия:

- `Применить`;
- `Сбросить`;
- `Обновить`;
- `Экспорт в XLSX`.

Кнопка `Обновить` повторно запрашивает данные с текущими фильтрами.

Автоматическое обновление в реальном времени в MVP не требуется.

## Состояния интерфейса

Интерфейс поддерживает:

- загрузку;
- успешное отображение;
- пустой результат;
- ошибку получения данных;
- ошибку формирования XLSX.

Примеры сообщений:

```text
По выбранным условиям движения материалов не найдены.
```

```text
Не удалось получить данные. Повторите обновление отчёта.
```

Строки отчёта доступны только для просмотра.

# Пагинация и производительность

Размер страницы по умолчанию:

```text
100 записей
```

Допустимые размеры страницы:

- 50;
- 100;
- 200.

Фильтрация, сортировка, пагинация и агрегирование выполняются на стороне Warehouse или базы данных.

Загрузка всех строк в память для отображения запрещена.

Сводные показатели рассчитываются по всей выборке, а не по текущей странице.

Основной поток JavaFX не должен блокироваться во время загрузки данных или формирования XLSX.

Новое применение фильтров должно отменять либо игнорировать результат предыдущего незавершённого запроса.

Жёсткое ограничение максимального периода в MVP не устанавливается.

Analytics не использует постоянный кэш отчётов.

Ошибка Analytics не должна:

- блокировать Warehouse;
- препятствовать проведению складских документов;
- изменять остатки;
- нарушать запуск других Capability.

# Права доступа

Analytics объявляет права:

```text
analytics.material-movements.view
analytics.material-movements.export
```

`analytics.material-movements.view` разрешает:

- видеть раздел Analytics;
- открывать отчёт;
- получать данные через Public API Analytics.

`analytics.material-movements.export` разрешает экспорт в XLSX.

Для экспорта пользователь также должен иметь право просмотра.

В MVP ограничения доступа по отдельным складам не применяются.

Пользователь с правом просмотра видит движения по всем складам.

## Рекомендуемые шаблоны ролей

| Роль | Просмотр | Экспорт |
|---|---:|---:|
| Руководитель | Да | Да |
| Начальник производства | Да | Да |
| Кладовщик | Да | Да |
| Производственный мастер | Да | Да |
| Оператор производства | Да | Да |

Роли являются шаблонами Security. Analytics не определяет и не хранит роли самостоятельно.

## Переходы к связанным объектам

Из отчёта допускается переход к:

- складскому документу;
- карточке материала;
- Customer Order;
- Order Item.

Переход разрешён только при наличии права просмотра соответствующего объекта.

При отсутствии права:

- значение остаётся видимым;
- ссылка неактивна;
- ошибка доступа не отображается.

# Аудит

Analytics не создаёт собственные записи аудита.

Не фиксируются:

- открытие отчёта;
- изменение фильтров;
- экспорт отчёта;
- дата и время экспорта;
- пользователь, выполнивший экспорт;
- применённые фильтры;
- формат файла.

Исходные складские операции и проведённые документы учитываются в аудите Warehouse и Document Engine в соответствии с их спецификациями.

# Экспорт XLSX

В MVP поддерживается только формат:

```text
XLSX
```

CSV и PDF не поддерживаются.

## Состав файла

Файл содержит:

- применённые фильтры;
- дату формирования;
- сводные показатели;
- полную таблицу движений по выбранным условиям.

Экспортируются все найденные строки, а не только текущая страница.

## Колонки экспорта

- дата и время;
- категория движения;
- тип операции;
- код материала;
- наименование материала;
- цвет;
- количество;
- единица измерения;
- склад-источник;
- склад-получатель;
- документ-основание;
- Customer Order;
- Order Item;
- пользователь.

Технические идентификаторы и служебные поля не экспортируются.

## Имя файла

```text
material-movements_YYYY-MM-DD_HH-mm.xlsx
```

## Правила

Если данные отсутствуют:

- файл не создаётся;
- отображается сообщение:

```text
По выбранным условиям данные отсутствуют.
```

Экспорт выполняется синхронно.

Фоновая очередь, отложенное формирование и уведомления о готовности не реализуются.

Строки XLSX должны записываться последовательно без предварительной загрузки всего результата в память.

# Public API Analytics

Analytics предоставляет минимальный read-only Public API.

## Получение движений

```java
MaterialMovementPage findMaterialMovements(
    MaterialMovementFilter filter,
    PageRequest pageRequest
);
```

## Получение сводки

```java
MaterialMovementSummary getMaterialMovementSummary(
    MaterialMovementFilter filter
);
```

## Получение деталей движения

```java
MaterialMovementDetails getMaterialMovementDetails(
    WarehouseMovementId movementId
);
```

## Формирование XLSX

```java
AnalyticsExportFile exportMaterialMovements(
    MaterialMovementFilter filter
);
```

## Фильтр

```java
public record MaterialMovementFilter(
    LocalDateTime periodFrom,
    LocalDateTime periodTo,
    WarehouseId warehouseId,
    MaterialId materialId,
    MovementCategory movementCategory,
    CustomerOrderId customerOrderId,
    OrderItemId orderItemId
) {}
```

Обязательные поля:

- `periodFrom`;
- `periodTo`.

## Категории

```java
public enum MovementCategory {
    RECEIPT,
    CONSUMPTION,
    TRANSFER,
    STATE_CHANGE
}
```

Категория Analytics не заменяет исходный тип Warehouse Operation.

## Domain Events

В версии 1.0 Analytics:

- не получает данные через Domain Events;
- не создаёт собственную аналитическую проекцию;
- не публикует собственные Domain Events.

Основным источником данных является Warehouse read-only Public API.

Переход к event-based read model допускается только в будущей версии после отдельного архитектурного решения.

# Внутренняя модель

Analytics не создаёт собственные бизнес-сущности и постоянные аналитические таблицы.

## Строка отчёта

`MaterialMovementRow` содержит:

- идентификатор движения;
- дату и время;
- категорию;
- тип Warehouse Operation;
- материал;
- цвет;
- количество;
- единицу измерения;
- склад-источник;
- склад-получатель;
- документ-основание;
- Customer Order;
- Order Item;
- пользователя.

## Сводка по единице измерения

```java
public record MaterialMovementQuantitySummary(
    UnitOfMeasure unit,
    BigDecimal openingBalance,
    BigDecimal receipt,
    BigDecimal consumption,
    BigDecimal transfer,
    BigDecimal closingBalance
) {}
```

## Общая сводка

```java
public record MaterialMovementSummary(
    List<MaterialMovementQuantitySummary> quantities,
    long operationCount
) {}
```

## Правила модели

- Для количества используется `BigDecimal`.
- Analytics не изменяет точность исходных данных.
- Округление определяется Warehouse и базовой единицей материала.
- Одна запись Warehouse Movement соответствует одной строке отчёта.
- Исторические движения не объединяются и не скрываются.
- Агрегация применяется только в сводной части.
- Собственные миграции аналитического хранилища в MVP отсутствуют.

# Тестирование

## Обязательные сценарии

Необходимо проверить:

- приход;
- расход;
- внутреннее перемещение;
- резервирование;
- снятие резерва;
- положительную корректировку;
- отрицательную корректировку;
- списание;
- результат инвентаризации.

Каждая операция должна иметь правильную категорию Analytics и исходный тип Warehouse Operation.

## Фильтры

Проверяются:

- период;
- склад;
- материал;
- категория;
- Customer Order;
- Order Item;
- сброс значений к состоянию по умолчанию.

## Расчёты

Проверяется, что:

- остаток на конец рассчитан корректно;
- межскладское перемещение не изменяет общий остаток;
- перемещение изменяет остатки отдельных складов;
- резервирование не изменяет физический остаток;
- разные единицы измерения не суммируются;
- корректирующее движение не удаляет исходное.

## Пагинация

Проверяются:

- размер страницы по умолчанию;
- переключение страниц;
- отсутствие повторов;
- отсутствие пропусков;
- сохранение фильтров;
- сводка по всей выборке.

## Права

Пользователь без `analytics.material-movements.view`:

- не видит раздел Analytics;
- не получает данные Public API.

Пользователь без `analytics.material-movements.export`:

- может просматривать отчёт;
- не может экспортировать XLSX.

## Экспорт

Проверяется, что XLSX:

- использует применённые фильтры;
- содержит все найденные строки;
- не ограничивается текущей страницей;
- содержит сводные показатели;
- сохраняет точность количества;
- разделяет единицы измерения;
- не содержит технических идентификаторов;
- не создаётся при отсутствии данных.

## Интеграционный сценарий

1. Warehouse создаёт движение.
2. Пользователь обновляет отчёт.
3. Движение появляется в Analytics.
4. Количество и направление совпадают с Warehouse.
5. Сводные показатели изменяются корректно.

# Критерии готовности MVP

Analytics считается готовым, если:

- реализован экран «Движение материалов»;
- работают согласованные фильтры;
- отображаются согласованные категории операций;
- корректно рассчитываются остатки, приход и расход;
- работает серверная пагинация;
- реализован просмотр деталей;
- работают права просмотра и экспорта;
- формируется XLSX;
- Analytics не изменяет данные Warehouse;
- отсутствуют собственные постоянные аналитические таблицы;
- автоматические тесты проходят;
- отсутствуют критические и блокирующие дефекты.

---

# Не входит в MVP

- финансовая стоимость материалов;
- себестоимость продукции;
- анализ производительности сотрудников;
- анализ сроков производства;
- аналитика Customer Order;
- загрузка производства;
- эффективность раскроя;
- отходы и коэффициенты использования;
- плановый и фактический расход;
- прогноз потребности;
- прогноз остатков;
- минимальные складские запасы;
- уведомления о дефиците;
- диаграммы;
- дашборды;
- сравнение периодов;
- пользовательский конструктор отчётов;
- сохранённые фильтры;
- автоматическая рассылка;
- CSV;
- PDF;
- отдельное аналитическое хранилище;
- аудит действий внутри Analytics.

---

# Возможное развитие

В следующих версиях могут быть добавлены:

- отчёт по текущим остаткам;
- расход материалов по Customer Order и Order Item;
- сравнение планового и фактического расхода;
- аналитика Cutting Plan;
- аналитика отходов;
- производственная аналитика;
- сравнение периодов;
- графики и дашборды;
- собственная Analytics Read Model.

Собственная read model может быть введена только после подтверждения проблем производительности на реальных данных.

Расширение Analytics не должно:

- изменять бизнес-логику других Capability;
- предоставлять право изменять доменные данные;
- создавать зависимость доменных модулей от Analytics;
- превращать Analytics в источник истины;
- нарушать Public API других Capability.

---

# Итоговый принцип

> Analytics должен предоставлять понятную информацию о движении материалов и не усложнять архитектуру системы до появления реальной необходимости.
